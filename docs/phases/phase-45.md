# Phase 45 -- v0.54.0: machine-first launcher, Docker publishing, editor polish, and chan open routing

Branch `release-0.54.0`, based on the v0.53.1 GA tag. It was cut from the v0.53.1 rc tip -- so it carried the merged bracketed-label grammar fork and the OSC 52 clipboard work in flight at the time -- and rebased onto v0.53.1 GA once that shipped. A feature round run as an orchestrator plus file-local subagent lanes: the launcher, the Docker CI path, the editor (two file-disjoint asks), and a follow-on `chan open` routing lane inherited from the v0.53.1 work. Two light design gates (the Docker publish policy and the launcher UX) folded into a single host survey, and each lane's diff was reviewed by adversarial sub-agents before the integrated gate. The dev host can build neither the Tauri desktop (no GTK/webkit) nor containers (no Docker) nor Windows, so those paths validate on the non-publishing `release.yml` dry-run build plus on-device; the editor, the launcher web build, and the `chan open` routing logic gate locally (`make web-check` / cargo). Per the project's release flow the rc was a `publish=false` dry-run build (there is no rc tag), and GA follows on-device validation.

## Theme

Reorganize the chan-desktop launcher machine-first, publish container images from the release, let an inline-code file link be re-pointed in place, move the ambient status notification clear of the terminal prompt, and make `chan open` serve where its shell actually runs.

## What landed (by lane)

### Launcher -- machine-first reorganization (ask 1)

The launcher's flat window feed is replaced by a machine-first tree: the local machine and each registered dev server render as equal top-level blocks, and each block lists its windows control-terminal-first, then standalone terminals, then the windows of each workspace nested inside a collapsible workspace card. The single top-bar `[+]` chooser is decoupled into a bottom-docked dashed "Add dev server" button and a per-block new-workspace action; bulk selection moved behind a Gmail-style Select toggle (checkboxes hidden until toggled or something is selected) with a bottom-docked bulk bar over dev servers and workspaces. A dev server whose control process disconnects now flashes an inline amber "reconnecting" cue on the control-terminal row instead of popping a survey modal -- the modal and its open-state are removed, the attention flash is kept. The local new-workspace form ships path-only; a settable display name is deferred. Read-only parity holds on the dev server and gateway surfaces, and workspace cards lift on hover. The reorg is pure-SPA with no `desktop/src-tauri` change; the component tree gained `WindowRow.svelte` and a pure `machineTree.ts`, and the flat `WindowFeed` and `ControlClosedSurvey` are deleted.

### Docker -- image publishing from the release (ask 2)

`release.yml` gained a `docker-publish` path that builds the four headless images -- `chan` plus the `identity`, `profile`, and `devserver-proxy` gateway services -- on a native-runner matrix (amd64 on `ubuntu-latest`, arm64 on `ubuntu-24.04-arm`), pushing each architecture by digest and assembling a multi-arch manifest. It reuses the existing `packaging/docker` Dockerfiles from a repo-root context with no Dockerfile edits, publishes under the `chanapp` Docker Hub namespace, and tags an immutable `X.Y.Z` on every release with `latest` reserved for GA; prerelease `-rc` tags push immutable images only. The jobs keep `contents: read` (Docker Hub login needs no GitHub token scope), gate every push on the release `context` job's `publish` output, and on a `publish=false` dry-run build each image without a registry. The path is a parallel sibling of `publish-release`, so a Docker failure never un-publishes the GitHub Release.

### Editor -- in-place inline-code link change and the ambient notification (asks 3 + 4)

E1 completes the inline-code file link from v0.53.0: with detect-and-open already shipped, typing inside a recognized link now opens the file picker (a new "code" template mode of the existing wiki picker) to re-point the target in place, percent-encoding the path so a spaced filename round-trips. A position-mapped armed-region latch opens the picker only on a resolved real-file link yet keeps it open while the token is edited through non-resolving intermediates; the change builds on the merged ref-aware parser without re-forking the grammar and leaves the `@lezer/markdown` pin untouched. Ask 4 moves the ambient status bubble from the bottom-left to the top-right, flips its collapse control to the right, and re-anchors the transfer bubble beneath it so transfers stack downward, clear of the terminal prompt; the session-handover and survey overlays are unchanged.

### chan open -- route by shell parentage (carried in from the v0.53.1 follow-up)

`chan open <path>` previously always tried the chan-desktop handoff before the local dev server, so a dev-server shell (an ssh session attached to a `chan devserver`) would open the workspace on the desktop GUI instead of the dev server it runs on. It now resolves a single serving target: an explicit `--standalone` / `--desktop` / `--devserver` flag (mutually exclusive) wins; otherwise it detects the shell's parent -- a chan-desktop or dev-server control socket exported into chan-spawned shells (`$CHAN_CONTROL_SOCKET`), resolved to its kind by the same machinery `chan ps` uses -- and serves there, falling back to standalone when no parent is detectable. A live dev-server parentage beats a leaked `CHAN_DESKTOP_HANDOFF`; `--devserver` inside a dev server is refused (no nested dev servers). The parentage probe is timeout-bounded so a wedged control socket cannot hang `chan open`, the route decision runs before any directory is created, and a `WorkspaceLocked` standalone open now points the user at `--devserver`. The routing logic is pure and unit-tested and was exercised end-to-end on Linux against a real dev server; the Windows handoff is best-effort this round (the Windows binary is not yet signable) and validates on-device later.

## Notes

- **Host-validated build.** The dev host has neither GTK/webkit, Docker, nor Windows, so the launcher (a WKWebView surface), the four container images, and the Windows `chan open` handoff validate on the cross-OS `release.yml` dry-run build and on-device; the editor, the launcher web build, and the `chan open` routing logic gate locally with `make web-check` / cargo. The integrated gate ran the host-runnable pre-push subset; the chan-desktop Tauri compile defers to CI.
- **Prerelease handling.** A new `prerelease` output on the release `context` job marks `-rc` tags as GitHub prereleases and keeps both the moving `latest` image tag and the GitHub "latest release" GA-only.
- **Adversarial review.** Each lane's diff was reviewed by independent sub-agents before the integrated gate -- the launcher's read-only parity and the flash-only deletion, the Docker dry-run safety and tag map, the editor's commit-roundtrip (which caught a document-corruption bug when click-jumping between two inline-code links on one line), and the `chan open` routing (which caught an unbounded parentage probe and a mkdir-before-refusal).
- **Deferred to v0.55.0.** A settable display name for local workspaces, the `packaging/kube` image-ref namespace prefix, the Windows terminal-idle reconnect and the `chan ps` extended-path fix, and all gateway/devserver backend work (tunnel-offline status, OAuth onboarding, multi-devserver).
