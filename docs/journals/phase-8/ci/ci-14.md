# ci-14 — Linux binaries on v0.12.0+ tags (chan CLI + chan-desktop on GitHub Releases)

Owner: @@CI
Cut: 2026-05-22 by @@Architect
Status: dispatched

## Goal

On the next `chan-v*` release tag (v0.12.0 or whichever
ships next), the GitHub Release page carries downloadable
Linux binaries alongside the macOS DMG:

* **chan CLI**: `.deb` / `.rpm` / `.tar.gz` per the
  `release.yml` matrix.
* **chan-desktop**: `.deb` / `.AppImage` per the
  `release-desktop.yml` Linux build job, uploaded to
  the GH Release (currently uploaded as workflow
  artifact only).

## Reference

[`../phase-8-bugs.md`](../phase-8-bugs.md) "Linux
binaries shipped on phase-8 next-release tags (chan CLI
+ chan-desktop)" — full bug body with the state-today
audit + the architecture caveat (aarch64 + x86_64 both
needed; local sdme + lima covers aarch64 fast-loop;
CI's `ubuntu-latest` is x86_64).

## Scope

### `release.yml` audit + matrix completion

1. Audit current matrix — confirm Linux x86_64 targets
   present + producing artifacts.
2. If Linux targets missing or producing wrong shapes:
   add. `.deb` + `.rpm` + `.tar.gz` for chan CLI on
   ubuntu-latest is the canonical shape.
3. Verify upload-to-release step picks up the Linux
   artifacts.

### `release-desktop.yml` upload-to-release extension

Per the `ci-7` audit trail: `release-desktop.yml`
builds Linux `.deb` / `.AppImage` on macos-latest jobs
+ uploads as workflow artifact
`chan-desktop-linux-x86_64-unsigned`. The
upload-to-release step apparently handles only macOS
DMG today.

Extend the release-job step (or add a new step) to wire
the Linux artifacts (`.deb` + `.AppImage`) onto the GH
Release alongside the DMG.

### Linux signing — out of scope

Linux binaries ship UNSIGNED for the v0.12.0+ dogfood
window (no equivalent of Apple Developer ID at this
phase). Linux signing options (dpkg-sig for .deb,
AppImage bundling key) exist but are Round-3 polish.
Document the unsigned status in release-notes copy when
v0.12.0 cuts.

### aarch64 vs x86_64 (release matrix)

Today's `release-desktop.yml` builds x86_64-only Linux
(per `ci-7` audit). aarch64 Linux release builds are a
Round-3+ matrix expansion. Note in the task tail; do
NOT expand the matrix in this task — that's a separate
scope decision.

## Acceptance

1. **release.yml fires on a `chan-v*` tag** + produces
   Linux CLI artifacts.
2. **release-desktop.yml uploads Linux artifacts to GH
   Release** alongside the macOS DMG. Both x86_64.
3. **Smoke validation**: push to a smoke branch +
   `gh workflow run release.yml --ref <smoke-branch>`
   (with the workflow_dispatch trigger from `ci-12`'s

   the run's output. NOTE: smoke validation against a
   real `chan-v*` tag is gated on @@Alex's release-cut
   beat; per-PR smoke can only validate the workflow
   YAML structurally.
4. **No regression** on the macOS DMG path.
