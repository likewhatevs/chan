### Audit findings (3 bugs in scope)

The task's "audit + complete" scope surfaced THREE
release-job bugs, all on the `chan-v*` tag path:

#### Bug 1 — release.yml release job's `if:` skips on chan-v* tags

`.github/workflows/release.yml` line 347 (pre-patch):

```yaml
release:
  name: github release
  if: startsWith(github.ref, 'refs/tags/v')
```

For a `chan-v0.12.0` tag push, `github.ref` is
`refs/tags/chan-v0.12.0`. `startsWith(..., 'refs/tags/v')`
returns **false** because the leading character is `c`,
not `v`. ci-11 added `chan-v*` to the tag trigger glob
(lines 33-37) but missed this gate, so on every phase-8
release tag the BUILD job has been running (matrix matches
the trigger) but the RELEASE job has been silently skipping.

Why we didn't see it before: phase-8's chan-v0.11.{0,1}
release.yml runs were billing-blocked; chan-v0.11.2's
release-desktop.yml fired green and shipped the DMG, so
"chan CLI binaries didn't reach the GH Release" got
attributed to the prior trigger bug (now fixed by ci-11)
rather than this second-layer release-job gate bug.

**Fix**: extend the `if:` to also match `refs/tags/chan-v`:

```yaml
if: startsWith(github.ref, 'refs/tags/v') || startsWith(github.ref, 'refs/tags/chan-v')
```

#### Bug 2 — release.yml VERSION strip produces "chan-v0.12.0" not "0.12.0"

`.github/workflows/release.yml` line 372 (pre-patch):

```bash
tag=${GITHUB_REF#refs/tags/}     # = chan-v0.12.0
version=${tag#v}                  # = chan-v0.12.0 (leading is `c`, not `v`)
printf '%s\n' "$version" > release-extras/VERSION
```

The `${tag#v}` parameter expansion strips ONLY a leading
`v`. For a `chan-v*` tag, the prefix is `chan-v` so the
strip is a no-op. The resulting VERSION file would
contain `chan-v0.12.0` instead of `0.12.0`. That breaks
the `chan upgrade` self-upgrade contract (it reads
`chan.app/dl/latest/VERSION` and expects a semver to
compare against the current binary's version string).

This bug is masked today by Bug 1 (release job skips on
chan-v* tags entirely; VERSION file never gets staged).
Bug 1's fix would unmask Bug 2; both must land together
to avoid shipping a broken VERSION file on the next
chan-v* tag.

**Fix**: strip both prefixes in order:

```bash
version=${tag#chan-}
version=${version#v}
```

For `v0.11.0`: `chan-` strip is no-op → `v0.11.0`; then
`v` strip → `0.11.0`. ✓
For `chan-v0.12.0`: `chan-` strip → `v0.12.0`; then `v`
strip → `0.12.0`. ✓
Both legacy and phase-8 conventions land the same shape.

#### Bug 3 — release-desktop.yml release job downloads + uploads only macOS

`.github/workflows/release-desktop.yml` release job
(pre-patch lines 394-410):

* Download: `pattern: chan-desktop-macos-*-signed`, which
  excludes the Linux artifact (`chan-desktop-linux-x86_64-unsigned`).
* Upload: `files: artifacts/**/*.dmg` ships only the
  DMG; `.deb` and `.AppImage` from the Linux build never
  reached the GH Release.

This is the canonical "chan-desktop Linux binaries don't
ship" gap that the bug-list entry calls out.

**Fix**:
* Broaden the download pattern to `chan-desktop-*` (catches
  both macOS and Linux artifacts; `notarization-diagnostics-*`
  artifacts on failure don't match this prefix; and
  `needs: build` implies the release job only runs on
  build success anyway).
* Add `.deb` + `.AppImage` to the upload `files:` list.
