# ci-11 — release.yml trigger-glob fix (chan CLI binaries on `chan-v*` tags)

Owner: @@CI
Cut: 2026-05-21 by @@Architect
Status: dispatched

## Goal

Restore intended behaviour for `release.yml` so chan CLI
binaries (.deb / .rpm / .pkg / .tar.gz) actually ship on
`chan-v*` release tags. Today the workflow's trigger
glob is `tags: ['v*']`, which doesn't match the phase-8
`chan-v*` convention, so the workflow has been silently
inert across chan-v0.11.0, chan-v0.11.1, chan-v0.11.2.

## Background

Surfaced 2026-05-21 by @@CI during v0.11.2's GH Release
asset verification (see
[`../alex/event-ci-architect.md`](../alex/event-ci-architect.md)
"Finding: `release.yml` trigger glob mismatch ...").
Empirical confirmation in the same poke:

* `gh release view chan-v0.11.{0,1,2}` → first two
  "release not found"; v0.11.2 has only the DMG.
* `gh run list --workflow=release.yml --limit=5` shows
  only `v0.11.0` + `v0.10.1` runs (both failed at
  billing). No `chan-v*` runs of `release.yml` ever
  fired.

Masking factors:

* Earlier phase-8 tags' `release-desktop.yml` runs were
  billing-blocked, so the absence of `release.yml`
  runs didn't surface either.
* `architect/journal.md`'s system description ("on the
  `chan-v*` tag per `release.yml`") matches the
  INTENDED behaviour, not the current state — so the
  mental model masked the gap.

## 2026-05-21 — Smoke dispatch result + workspace-wide gap finding

Smoke dispatch run [`26227752597`](https://github.com/fiorix/chan/actions/runs/26227752597)
completed: red. `clippy + test (ubuntu-latest)` failed
at `cargo clippy --all-targets` compilation of
`glib-sys v0.18.1`, dying on `pkg-config` not finding
`glib-2.0`. The rest of the chain (test-macos →
5-target build → release) was skipped per `needs:`
gating.

Per @@Architect's "if the dispatch reds, that's an
out-of-lane bug routing" framing, this is NOT a ci-11
blocker. Investigating further revealed a much
larger latent gap:

**Every ci.yml run on main for the last 15 commits
has failed with the SAME glib-sys / glib-2.0 root
cause.** The per-PR CI gate has been silently broken
since at least 2026-05-19. `gh run list --workflow=ci.yml
--limit=15` shows 15 / 15 failures; sampled runs
(`e7468db` post-v0.11.2 close-out; `60901c1` chan
v0.11.2 tag commit) confirm the same `glib-sys` root
cause.

Root cause: `desktop/src-tauri` is a workspace member
(per `Cargo.toml`); `cargo clippy --all-targets` from
the workspace root touches all members including
chan-desktop's Tauri deps; the GTK system libs needed
to compile `glib-sys`'s build script aren't apt-
installed in ci.yml's or release.yml's test jobs.
Only `release-desktop.yml` installs them. Local devs
have GTK pre-installed for `make run`, so the issue
is invisible locally and the pre-push hook passes
clean.

Two fix shapes surfaced for architect routing:

* **(a)** Apt-install the GTK stack in ci.yml +
  release.yml test jobs (mirror release-desktop.yml's
  pattern). Correct + heavier (4 jobs need it; Windows
  may need chocolatey or may not need the GTK stack
  at all, needs verification).
* **(b)** Exclude `desktop/src-tauri` from the
  workspace clippy/test sweep via
  `--exclude chan-desktop`. One-line change in ci.yml
  + release.yml. Trade-off: chan-desktop drops out of
  the per-PR clippy gate, but as noted, the gate was
  never actually catching anything because of this
  same bug. Faster fix; matches the current effective
  state.

Recommendation tilt: **(b)** for the immediate
unblock; (a) as a follow-up if/when the workspace
policy is "all members pass per-PR clippy".
