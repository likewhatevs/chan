## Bug 2 — `chan close {path}` / `--remove` never hands off to chan-desktop
**Risk: med · Effort: M · Confidence: 0.70 · device-validate: yes**

**Root cause (verified).** There is **no close handoff**. `cmd_close`/`cmd_remove`
(`crates/chan/src/lib.rs`) tear down only through the per-pid control socket
(`unserve_running` → `control_socket_for_pid` → `ControlRequest::Close`), while `open`
uses the well-known per-user **handoff** socket. The handoff `Request` enum
(`crates/chan-server/src/handoff.rs`) and the desktop listener
(`desktop/src-tauri/src/main.rs`) have **no Close arm**. The control-socket close *does*
reach the embedded host and the native window *does* close via the watcher — the narrow
defect is that the desktop's runtime map `AppState.serves` is **never updated** by that
path, so (a) the launcher shows the workspace stale-on, (b) `persist_workspaces`
resurrects it on next boot from the stale map, and (c) a later `chan open` of the same
path mints a token-less dead window.

**Fix.** Add `Request::CloseWorkspace { workspace_path, remove }` + `Response::Closed` +
a `try_close_workspace` client mirroring `try_handoff` (unix UDS / windows named pipe).
Thread `personality` into `cmd_close`/`cmd_remove` (incl. the `run()` dispatch arms,
which currently drop it); under Desktop personality attempt the close handoff first,
falling back to the existing control-socket path on any non-HandedOff outcome (preserves
devserver/standalone + old-desktop behavior). Desktop listener: add a `CloseWorkspace`
arm running the same teardown the UI uses — `serve::stop` for close, plus the registry
unregister for `remove` (factor a helper out of the `remove_workspace` Tauri command) —
then `persist_workspaces`.

**Touch:** `crates/chan-server/src/handoff.rs`, `crates/chan/src/lib.rs`,
`desktop/src-tauri/src/main.rs`, `desktop/src-tauri/src/serve.rs`. **Note:** the literal
"never closes" may *also* be a `control_socket_for_pid` discovery failure (a GUI-launched
desktop whose `TMPDIR` differs from the terminal's); the handoff fix sidesteps it.

## Bug 3 — Retry/reconnect overlay: cmd+\` passthrough · Abandon · spinner
**Risk: med · Effort: L · Confidence: 0.78 · device-validate: yes (load-bearing)**

**Root cause (verified).** Two gaps. (a) `KEY_BRIDGE_JS` deliberately does **not** claim
`Backquote`; the swallow is the **SPA** — `App.svelte`'s bubble-phase `onWindowKey`
blanket-`preventDefault()`s every key while `ui.disconnectBlocking`. (b) The overlay
(`DisconnectOverlay.svelte`) renders only `Retry`; all teardown + spinner machinery
exists but is unreachable (`teardown_devserver_connection` in `main.rs`; the launcher's
`disconnectDevserver` in `library.svelte.ts` *is* the required spinner lifecycle; the
"Abandon = disconnect" contract is ratified).

**Fix.** (a) In `onWindowKey`'s `disconnectBlocking` branch, `if (e.code ===
'Backquote') return;` before the blanket swallow (mac-relevant only). (b) Add an
`Abandon` button gated on `windowLibraryId() !== 'local' && isTauriDesktop()`, invoking a
new IPC `abandon_devserver_for_window` → emit `devserver-abandon` → launcher
`disconnectDevserver`. **Corrections from verify (must apply):** resolve the devserver id
via connected `state.devservers` + the **cached** `library_id_of` (NOT
`devserver_id_for_library`, which returns None exactly when the overlay shows);
`show_window("main")` before emitting (the launcher hides, not destroys); the window
closes async via the watcher; **only `permissions/app.toml`** needs the ACL grant (not
`capabilities/workspace.json`); non-loopback devserver windows have no IPC so Abandon is
inert there (document).

**Touch:** `web/packages/workspace-app/src/{App.svelte,components/DisconnectOverlay.svelte,api/desktop.ts}`,
`web/packages/launcher/src/App.svelte`, `desktop/src-tauri/src/main.rs`,
`desktop/src-tauri/permissions/app.toml`. The load-bearing unknown: that a
preventDefault'd cmd+\` in WKWebView actually blocks AppKit window cycling — confirm on
device.

## Bug 4 — `cs open file.md` cursor must land at row 1, col 1
**Risk: low–med · Effort: M · Confidence: 0.82 · device-validate: yes**

**Root cause (verified).** The caret is a **mount-time-only** input: `maybeRestoreCaret`
runs once, defaults an absent caret to `{0,0}`, and **latches**. A clean **first** open
already lands at (0,0) — the suspected render race is **not** the repro. The real bug:
the explicit-open path expresses **no caret intent** and has **no live channel** to an
already-mounted editor. Panes keep one `FileEditorTab` per tab alive
(`Pane.svelte`), so reopening hits the existing-tab branch (`tabs.svelte.ts`) on a
latched editor — nothing re-places the caret. **Dominant repro: reopen an already-open /
kept-alive tab** (File-Browser click A→B→A, or `cs open A` twice); session-restore is the
same branch; first-open is not a repro. A caller-only `{0,0}` would be a **near-no-op**
for the symptom — the live re-drive is the whole fix.

**Fix.** (1) Add `landAtTop` to `OpenFileOptions`, passed from the **explicit-open
callers only** — `open_file` (`store.svelte.ts`), `FileTree.svelte`,
`FileBrowserSurface.svelte`, create/duplicate — **not** search/wikilink/mention. (2) Add
a live **per-tab caret-command signal** (module-level, NOT on `FileTab` or it leaks into
serialize), bumped in `openInPane` on all branches. (3) Consume it in a `FileEditorTab`
effect. (4) Add `resetCaret(from,to)` to both editors = selection + **`scrollIntoView`**
+ focus (do **not** reuse `focusAt`, which omits the scroll → caret off-screen on a
scrolled reopen). (5) Keep the `{0,0}` fallback. **Descope** the don't-latch-on-partial
idea — separate bug.

**Touch:** `web/packages/workspace-app/src/state/{tabs,store}.svelte.ts`,
`components/{FileTree,FileBrowserSurface,FileEditorTab}.svelte`,
`editor/{Source,Wysiwyg}.svelte`, `editor/newFileCaret.test.ts`. **Decision (default
yes — the title says so):** an explicit open forces top even when reopening a
deliberately-scrolled tab; page-*reload* restore still keeps the remembered caret.

## Bug 5 — link autocomplete inside `[](url)` must offer the link itself first
**Risk: low · Effort: M · Confidence: 0.87 · device-validate: yes**

**Root cause (verified).** The `[label](url)` URL-slot autocomplete is the wiki bubble in
**raw mode**; `activeHits()` (`editor/bubbles/wiki.ts`) builds the list purely from
`/api/link-targets` search hits — no entry for the literal current URL, so cmd+enter can
never open it.

**Fix (all in `editor/bubbles/wiki.ts`).** Add a `SelfHit` sentinel; in raw+file mode
read the full slot literal `doc.sliceString(triggerStart, triggerEnd)`, resolve via
`parseInternalLink(literal, "", fromPath)` (already exported from
`editor/widgets/wikilink.ts`), and **prepend it as index 0**. Render an "OPEN <url>" row;
`openSelected()` handles it first → `onOpenLink(target, anchor)` (pill-click parity). The
`commit()` Self guard is **mandatory** (else `fileLinkInsert(undefined,…)` corrupts the
doc); route a **mouse click** on the Self row to open too. Clamp the selectedIndex resets
against `activeHits().length`. **Decisions:** plain-Enter on the pre-selected Self row
(keep-as-is vs commit top hit — a raw-mode behavior change); external URLs
(`http`/`mailto`/`#`) return null from `parseInternalLink` → no Self row and the current
open path can't open them (scope out or handle separately).

**Touch:** `web/packages/workspace-app/src/editor/bubbles/{wiki.ts,wikiLinkTargets.test.ts}`.
Precedent: `editor/bubbles/image.ts` already prepends a synthetic raw-mode row.

## Bug 6 — black horizontal bar at the bottom of the terminal
**Risk: low · Effort: S · Confidence: 0.86 · device-validate: yes (visual)**

**Root cause (verified).** `xterm.css` ships `.xterm .xterm-viewport {
background-color:#000 }` filling the full `height:100%` `.xterm` box, but xterm themes
**only the inner scrollable node** (sized to `rows*cellHeight`). The sub-cell remainder
below the last row shows the bare `#000` — a DOM background, renderer-agnostic.

**Fix (one line).** Add `background-color: var(--bg);` to the existing
`.terminal-host :global(.xterm-viewport)` rule in
`web/packages/workspace-app/src/components/TerminalTab.svelte`. The Svelte-scoped selector
(0,3,0) beats xterm's (0,2,0) — no `!important`.

**Touch:** `web/packages/workspace-app/src/components/TerminalTab.svelte` (style block only).

## Bug 7 — chan-desktop startup doesn't restore local workspace ON/OFF
**Risk: low · Effort: S · Confidence: 0.90 · device-validate: yes**

**Root cause (verified).** Two writers to the local overlay (`~/.chan/workspaces.json`).
The launcher toggles on/off **over HTTP** (`handle_workspace_on/off` → `set_overlay`,
keyed by root) — correct and immediate. But the desktop keeps a shadow `AppState.serves`
mutated **only** by `serve::start/stop`, never by the HTTP path; `persist_workspaces`
(`desktop/src-tauri/src/main.rs`) does `overlay.replace()` of the whole overlay from the
stale `serves` keys, and runs at `RunEvent::Exit` **before** `stop_all` — so every clean
quit overwrites the correct overlay with the stale shadow. Next boot reads `on_paths()`
and serves the wrong/empty set. (Devserver overlay is a separate, non-auto-restored store
— **exception preserved**.)

**Fix.** Rewrite `persist_workspaces` to reconcile against the host's **actual mounted
set**, mirroring `crates/chan-server/src/devserver.rs::persist_state`: iterate
`embedded.library().list_workspaces()`, filter by `host.is_root_mounted(&ws.root_path)`,
map to `PersistedWorkspace{ path, on:true }`, `overlay.replace(rows)`; stop reading
`state.serves`. Add a one-line `embedded.is_root_mounted` passthrough. Keep both call
sites (exit runs before `stop_all` while everything is still mounted).

**Touch:** `desktop/src-tauri/src/{main.rs,embedded.rs}`.

## Phase D — `chan serve` → terminology sweep (context-dependent)

`chan serve` was renamed; this is a **three-way** rewrite, not a `sed` (verified against
`crates/chan/src/lib.rs`: `Command::Open` carries `SERVE_LONG_ABOUT`; tunnel flags live on
`Command::Devserver`, not `Open`):

| Context | Replacement |
|---|---|
| Serve one local workspace (`chan serve <path>`, test servers, "where you ran `chan serve`") | **`chan open`** |
| Tunnel mode (`chan serve --tunnel-*`) + gateway "fleet of instances dialing in" | **`chan devserver`** (`chan open --tunnel-*` doesn't exist) |
| `CHANGELOG.md`, `docs/phases/phase-*.md`, `docs/journals/*` | **leave untouched** — they record the rename itself |

Also leave alone: `packaging/docker/{README,chan.Dockerfile}` (already say "there is no
`chan serve`") and every `chan server` (the *server*, not the command). ~28 hits map to
`chan open` (`.agents/*.md`, `crates/*/design.md`, `desktop/*.md`, the user-facing
`DisconnectOverlay`/`MissingTokenOverlay` strings); ~30 to `chan devserver` (`gateway/**`,
`SECURITY.md`, `.agents/principles.md`, `web/packages/profile/*`). The gateway/tunnel
batch is separable if a tighter cut is wanted.
