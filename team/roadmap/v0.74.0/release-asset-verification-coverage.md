# Release Asset Verifier Misses The Windows Artifacts

Status: accepted scope for v0.74.0. Small, concrete gap found during the v0.73.0 release.

## Problem

`web/packages/marketing/scripts/verify-release-assets.mjs` hand-maintains the list of assets a published GitHub Release must carry, and that list has no Windows entries. `publicAssets` (verify-release-assets.mjs:22-42) is three CLI tarballs (musl x86_64, musl aarch64, darwin aarch64), seven desktop packages (dmg, two AppImages, two debs, two rpms) and eight gateway debs (four services from `gatewayServices` times amd64/arm64). `updaterAssets` (verify-release-assets.mjs:43-46) adds the macOS updater payload and its detached `.sig`. `requiredAssets` is the concatenation (verify-release-assets.mjs:48) and every name missing from the release becomes an error (verify-release-assets.mjs:53-57). That is 20 required names. The v0.73.0 release carries 22 assets; the two the verifier never looks at are `chan-x86_64-pc-windows-msvc.zip` and `Chan_0.73.0_x64-setup.exe`.

Both Windows artifacts are produced on every release run. The `windows-artifacts` job (.github/workflows/release.yml:590-593) is gated only on `context`, and its build step zips the signed CLI as `chan-x86_64-pc-windows-msvc.zip` (.github/workflows/release.yml:691) and copies the signed NSIS installer as `Chan_${VERSION}_x64-setup.exe` (.github/workflows/release.yml:700), then uploads `release-artifacts/*` as the `release-windows` artifact with `if-no-files-found: error` (.github/workflows/release.yml:770-775).

`if-no-files-found: error` only catches a fully empty staging directory. If one of the two staging lines is dropped, renamed, or produces a version-mismatched filename, the artifact upload still succeeds, `softprops/action-gh-release` still uploads whatever arrived (.github/workflows/release.yml:816-823), and the verifier reports success. Downstream, `collect-release-assets.mjs` treats the Windows pair as optional (collect-release-assets.mjs:56-66) and `generate-release-metadata.mjs` only emits the `desktop-windows-nsis` and `cli-windows-x64` downloads when the manifest contains them (generate-release-metadata.mjs:153-169, 377). So the failure mode is silent: the release ships, the install page quietly loses its Windows buttons, and nothing goes red.

Two aggravating facts about when this code runs at all. First, the verifier executes as `npm run verify:release -- --tag "$TAG"` inside `publish-release` (.github/workflows/release.yml:825-830), and that job is gated on `if: needs.context.outputs.publish == 'true'` (.github/workflows/release.yml:778). `pages-artifact`, which runs `collect-release-assets.mjs` and `generate-release-metadata.mjs`, is gated the same way plus GA-only (.github/workflows/release.yml:845). A `publish=false` dry run therefore never executes any of the three scripts; their first real execution on any change is the GA run itself. Second, `make web-marketing-check` (Makefile:272-274) runs `npm run check -w @chan/marketing`, whose script (web/packages/marketing/package.json:20) gives `verify-release-assets.mjs` only `node --check` syntax validation. `collect` and `generate` do get behavioural fixture coverage there, via `smoke-release-assets-manifest.mjs` and `smoke-release-metadata.mjs`, and that coverage does include Windows (smoke-release-assets-manifest.mjs:46-57, smoke-release-metadata.mjs:63-91). The verifier is the one script in the trio with no behavioural test.

Note also that the required list is spelled out a third time in the fixture harness: `namesFor` and `optionalNames` in smoke-release-assets-manifest.mjs:243-257 duplicate the same names again. Only the gateway service names are single-sourced, parsed out of the Makefile's `GATEWAY_RELEASE_CRATES` (Makefile:57) by gateway-services.mjs:21-35.

## Desired Contract

The verifier requires every asset the release workflow produces, including the two Windows artifacts, and the required-name list stops being maintained in three parallel copies.

Honest assessment of "single source": deriving the list from `release.yml` itself is not practical. The workflow never enumerates its outputs; the names are produced by shell and PowerShell steps inside job bodies (release.yml:691, 700), by `cargo tauri build` bundle conventions, and by `cargo-deb`. There is no machine-readable manifest to read, and inventing one that the workflow actually consumes would mean rewriting every staging step in six jobs. That is a much larger change than the gap warrants.

The smallest robust alternative, in order of value:

1. A shared module in the marketing package, for example `scripts/release-assets.mjs`, exporting `requiredAssets(version)` and `updaterAssets(version)`, consumed by `verify-release-assets.mjs`, `collect-release-assets.mjs` and `smoke-release-assets-manifest.mjs`. This collapses the three hand-kept copies into one and follows the pattern gateway-services.mjs already established. It does not tie the list to the workflow, but it removes the drift that actually bites.
2. Move the Windows pair from optional to required in that shared module. `publish-release` already lists `windows-artifacts` in `needs` (.github/workflows/release.yml:786), so a skipped or failed Windows job blocks publication outright; there is no supported release shape that legitimately lacks Windows assets. `collect-release-assets.mjs` may keep them optional if the intent is to stay tolerant of the archived releases it walks with `--latest-count 5`, but that tolerance should be a deliberate, commented exception rather than the default the verifier inherits.
3. A workflow-coverage lint in the fixture smoke: for each required name that appears literally in a workflow staging step, assert the string is present in `.github/workflows/release.yml` after normalizing the version interpolation to a token. This catches "someone deleted or renamed the staging line" without needing a real release. Gateway debs and the tauri bundle outputs never appear literally in the workflow and must be exempted by name, with a comment saying why.
4. A local mode for the verifier. `collect-release-assets.mjs` already accepts `--release-json` and `--asset-dir` (collect-release-assets.mjs:104-120), which is exactly what lets `smoke-release-assets-manifest.mjs:230-298` drive it offline against a fixture. Giving `verify-release-assets.mjs` the same `--release-json` input, alongside its current `--tag` and `--skip-asset-url-heads` (verify-release-assets.mjs:109-127), is what makes the verifier testable in `make web-marketing-check` instead of only on a real GA.

## Acceptance

- `make web-marketing-check` passes with a new fixture smoke, `smoke-verify-release-assets.mjs`, wired into the marketing `check` script alongside the existing smokes (web/packages/marketing/package.json:20). It must assert both directions: a fixture release containing all required names exits 0, and a fixture release with `chan-x86_64-pc-windows-msvc.zip` removed exits non-zero with a `missing release asset:` line naming that file. Run the smoke with `--skip-asset-url-heads` so it makes no network calls.
- Red proof for the smoke: delete the two Windows entries from the shared `requiredAssets` list and re-run `npm run check -w @chan/marketing`. The negative case stops erroring and the smoke fails. Conversely, adding the Windows names to the fixture's complete-release case while leaving the missing-asset expectation in place must also fail.
- Red proof for the workflow-coverage lint: locally delete the zip staging line (.github/workflows/release.yml:691) and run `make web-marketing-check`. It must go red naming `chan-x86_64-pc-windows-msvc.zip`. Revert; the check goes green. This is the mutation that models the real failure, an edit to the Windows job that silently drops an artifact.
- `node scripts/verify-release-assets.mjs --tag v0.73.0` exits 0 against the real published release once the Windows names are required. This is a read-only GitHub API call and can be run by anyone with a token. Expect the two existing warnings (`VERSION asset absent`, `SHA256SUMS asset absent`); v0.73.0 publishes neither, and verify-release-assets.mjs:59-77 treats both as warnings, not errors.
- No usable historical negative case exists. v0.71.0, v0.72.0 and v0.73.0 all carry both Windows assets, so there is no past tag against which the strengthened verifier can be shown going red. The fixture smoke is the only automated red proof; do not claim otherwise in the commit message.
- Proving the end-to-end path, verifier failure actually failing the `publish-release` job on a live tag, requires a real GA run and is the owner's to observe. Name it as such: it cannot be exercised by a `publish=false` dry run, because `publish-release` is gated on `publish == 'true'` (.github/workflows/release.yml:778).

## Boundaries

- Do not restructure the release workflow to emit an asset manifest. That was assessed and rejected above; the shared JS module plus the coverage lint is the agreed shape.
- Do not change what `windows-artifacts` builds, signs, or names. This item is about verification only.
- Do not remove or weaken the existing `VERSION` and `SHA256SUMS` warning behaviour (verify-release-assets.mjs:59-77). Promoting those to errors is a separate decision with its own release consequences.
- Do not touch `gateway-services.mjs` or the `GATEWAY_RELEASE_CRATES` derivation (Makefile:57). It already works and is the model being copied.
- Keep the `--tag` and `--skip-asset-url-heads` flags and their current semantics intact; the workflow invocation at .github/workflows/release.yml:830 must keep working unchanged.
- If `collect-release-assets.mjs` keeps the Windows pair optional while the verifier makes it required, the divergence must be a one-line comment at collect-release-assets.mjs:56-66 explaining why, not an unexplained inconsistency between two files that otherwise share a module.
