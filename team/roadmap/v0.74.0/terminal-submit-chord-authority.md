# The Submit Chord Is Chosen By The Sender And Never Validated Against The Target

Status: accepted scope for v0.74.0. A confirmed defect with a live user report.

## Problem

`cs terminal write --submit=<agent>` names the chord the SENDER believes the target reads. Nothing on the write path compares that value against the target session's real agent, and there is no warning when they disagree.

`--submit` is a plain clap value on the writer's own command line (`crates/chan-shell/src/cli.rs:729-738`). The writer resolves it in its OWN process, against its OWN environment and config, before the value leaves the machine: `submit.map(ResolvedSubmit::resolve)` at `crates/chan-shell/src/cli.rs:1795`. The resolved spec rides the control wire as an opaque field (`crates/chan-shell/src/wire.rs:150-162`, `TermWrite`), and the server enqueues it verbatim onto every matched session: `term_write` at `crates/chan-server/src/control_socket.rs:3260-3287` calls `enqueue_write_matching` at `crates/chan-library/src/terminal_sessions.rs:1440-1479`, which passes `submit.cloned()` into each matched session's FIFO without consulting that session at all.

The server already holds the correct answer for every live session. Each session carries its spawn command and spawn environment (`crates/chan-library/src/terminal_sessions.rs:653-663`), and the server uses exactly those two inputs to compute the agent for the SPA session frame: `SubmitAgent::derive(session.spawn_command(), session.spawn_env("CHAN_AGENT"))` at `crates/chan-server/src/routes/terminal.rs:946-951`. That is why the browser Rich Prompt submits correctly into the very tab a peer poke cannot reach: the Rich Prompt path sends an agent NAME that the server resolves against the session (`crates/chan-server/src/routes/terminal.rs:146-151`, `crates/chan-server/src/routes/terminal.rs:726-734`), while the `cs` path sends fully-resolved bytes the server never questions.

A sender also has no runtime way to look up the answer. `cs terminal list --json` emits `name`, `session_id`, `window`, `window_kind`, `window_status`, `pane`, `tab`, `cwd`, `queue_depth` and nothing else (`crates/chan-server/src/control_socket.rs:3393-3403`), backed by `TerminalSessionSummary` (`crates/chan-library/src/terminal_sessions.rs:311-330`), which carries no agent field. The only published source of a peer's chord is the static roster generated into `bootstrap.md` at team creation (`crates/chan-server/src/routes/team_config.rs:122-125` for `member_agent`, rendered by `render_roster_table` at `crates/chan-server/src/routes/team_config.rs:663-729` and `render_poke_chords` at `crates/chan-server/src/routes/team_config.rs:735-748`, and instructed at `crates/chan-server/src/routes/team_config.rs:402-414`). A roster is a snapshot of team-creation config: it does not cover sessions created by hand, sessions whose command was overridden, or a tab that was restarted into a different agent.

The resolution rule the server would apply is `SubmitAgent::derive` at `crates/chan-shell/src/submit.rs:161-185`: `CHAN_AGENT` wins when it names a known agent or an explicit `none` / `shell`, an unrecognized value falls through to a LOOSE whole-word sniff of the command, and no match yields `None`, which means a shell member with no chord at all.

The failure is byte-level, and its shape is what proves the diagnosis. The built-in templates are at `crates/chan-shell/src/submit.rs:195-202`: Claude is `{}` plus a modifyOtherKeys CSI and no CR; Codex and OpenCode are BYTE-IDENTICAL bracketed paste plus CR; Gemini is a bare CR. So if every agent passes its OWN name as `--submit`, the broken pairs are exactly the ones that do not share bytes: claude to codex and claude to opencode fail in both directions, while codex to opencode works by accident because the two templates are the same string. Gemini is broken against all three in both directions, and for a reason the code already records: Gemini converts a Return that arrives too soon after inserted text into Shift+Return, including text delivered as bracketed paste, which is why its body and CR are split into separate writes (`crates/chan-shell/src/submit.rs:296-305`). This matches the user report precisely, including the pair they did not report as broken.

A `--tab-group` broadcast makes it structural rather than accidental. One `--submit` value fans out across every session matching the group (`crates/chan-library/src/terminal_sessions.rs:1446-1478`), so in a mixed-agent team a broadcast is guaranteed wrong for at least one member.

The symptom is quiet. When the chord does not fire, the text parks in the target's compose box, and the drainer releases the next queued message anyway once `WRITE_QUEUE_GEN_START_CAP_MS` elapses (`crates/chan-library/src/terminal_sessions.rs:64-70`), whose comment names this exact case: "a message that did not trigger generation (e.g. no submit chord) does not wedge the queue." Trailing newlines are trimmed on every delivery path (`crates/chan-shell/src/submit.rs:285`, `crates/chan-shell/src/submit.rs:303`, `crates/chan-shell/src/submit.rs:334`, `crates/chan-shell/src/submit.rs:358`, and in the batch framer at `crates/chan-library/src/terminal_sessions.rs:2150` and `crates/chan-library/src/terminal_sessions.rs:2201`), so successive pokes that arrive separately concatenate into the parked draft with NO separator. The sender sees `queued at position N` and a green exit either way (`crates/chan-server/src/control_socket.rs:3281-3287`).

One related trap belongs in the same fix, because it is the first thing an operator reaches for. `CHAN_MODE` is not a variable chan reads anywhere: a repository-wide grep over Rust, TypeScript, Svelte and Markdown returns no reader. Setting it is silently inert. The variable is `CHAN_AGENT` (`crates/chan-shell/src/submit.rs:161-172`, `crates/chan-server/src/routes/terminal.rs:949`, `crates/chan-server/src/routes/team_config.rs:123`).

## Desired contract

Two halves, both required. Authority fixes the defect; discovery removes the reason a sender was guessing.

**The server is authoritative over the chord on the write path.** At enqueue time, for each matched session, the server derives that session's agent from its own spawn command and `CHAN_AGENT` using the same `SubmitAgent::derive` the SPA session frame already uses, and delivers THAT agent's chord regardless of what the sender named. A session that derives to `None` (a shell member) receives no chord, even when the sender asked for one. A group broadcast therefore delivers a per-session-correct chord to a mixed-agent team instead of one guess for everybody.

The sender's `--submit` keeps exactly one meaning after this change: WHETHER to submit, not HOW. Present means "submit this into the target"; absent means "type it and leave it parked", which is the existing documented raw-write behaviour (`crates/chan-shell/src/help.rs:1359-1364`). The item must decide, and record, whether a disagreement between the sender's named agent and the derived agent is reported back in the control reply string (the reply already carries queue position and cap-drop text at `crates/chan-server/src/control_socket.rs:3281-3287`, so a divergence note has an obvious home) or accepted silently. Silence is only acceptable if discovery lands in the same release; a wrong-but-corrected value that nobody is told about will keep circulating through hand-written team docs.

The env-only template override must survive. Today the writer resolves the template in its own process precisely so a `CHAN_SUBMIT_<AGENT>` env value on the writer takes effect (`crates/chan-shell/src/cli.rs:1791-1795`, override order documented at `crates/chan-shell/src/submit.rs:207-213`). Moving authority to the server moves the override venue to the server's environment and its `<config>/chan/submit.toml`. That is a behaviour change on a documented surface and must be stated in the help text and in `crates/chan-shell/design.md`, not left to be discovered.

**A sender can discover a target's agent at runtime.** `cs terminal list --json` gains an `agent` field per session, carrying the same derived value (`"claude"` / `"codex"` / `"gemini"` / `"opencode"`, and null for a shell session with no chord). This requires an agent field on `TerminalSessionSummary` (`crates/chan-library/src/terminal_sessions.rs:311-330`), populated where the summary is built (`crates/chan-library/src/terminal_sessions.rs:1336-1348`), and rendered into the JSON entry (`crates/chan-server/src/control_socket.rs:3393-3403`). The human-readable `cs terminal list` output should show it too, since a poking agent reads that surface as often as the JSON one.

**Rich Prompt must not change.** The browser path already works and already goes through the server: `ClientFrame::Prompt` carries an agent NAME which the server resolves with `SubmitAgent::from_agent_name` (`crates/chan-server/src/routes/terminal.rs:726-734`), and the session frame reports the derived agent to the SPA (`crates/chan-server/src/routes/terminal.rs:946-960`). The wire shape of `ServerFrame::Session`, including `submit_agent`, is pinned by tests at `crates/chan-server/src/routes/terminal.rs:1428-1450` and `crates/chan-server/src/routes/terminal.rs:1504-1561`, and by the SPA-side wiring pin at `web/packages/workspace-app/src/components/richPromptTerminalWiring.test.ts:45-47`. This item touches neither the frame nor `submitMode.ts`. If the implementation makes the `cs` path share the Rich Prompt path's resolution point, that is a merge toward the working behaviour, not a change to it, and the existing pins must stay green untouched.

**The `CHAN_MODE` trap is documented.** Wherever `CHAN_AGENT` is taught (the `cs terminal write` long help at `crates/chan-shell/src/help.rs:1357+`, `crates/chan-shell/design.md:140`, and the team bootstrap text at `crates/chan-server/src/routes/team_config.rs:402-414`), state plainly that `CHAN_AGENT` is the variable and that `CHAN_MODE` is read by nothing.

## Rejected fix: appending a trailing newline

Appending `\n` to poke messages was considered and is rejected on both branches.

For a `--submit` message it is a pure no-op: the trailing newline is trimmed at six sites before the chord is applied (`crates/chan-shell/src/submit.rs:285`, `crates/chan-shell/src/submit.rs:303`, `crates/chan-shell/src/submit.rs:334`, `crates/chan-shell/src/submit.rs:358`, `crates/chan-library/src/terminal_sessions.rs:2150`, `crates/chan-library/src/terminal_sessions.rs:2201`), and deliberately so: a newline before the chord lands inside the agent's draft and splits the buffer (`crates/chan-shell/src/submit.rs:282-284`).

For a raw write it is worse than useless. `cs terminal write` documents "No newline is appended", with `ls` versus `$'ls\n'` as the worked example (`crates/chan-shell/src/help.rs:1359-1364`). Appending one would auto-execute half-typed shell commands in every shell tab, turning a typing primitive into an execution primitive. The trailing-newline trim is not the bug; the unvalidated chord is.

## Acceptance

Every check below is named with the deliberate mutation that must turn it red, per the standing rule recorded at `team/release/release-v0.73.0.md:37`.

1. **Server overrides a wrong sender chord.** A `chan-library` test enqueues with a Claude-resolved `submit` onto a session whose spawn command is `codex`, drains it, and asserts the delivered PTY bytes are the codex bracketed-paste-plus-CR form, not the Claude CSI. Red mutation: keep passing the caller's `ResolvedSubmit` through `enqueue_write_matching` unchanged.
2. **A shell target gets no chord.** Same harness, session spawned as `bash` (derives to `None` per `crates/chan-shell/src/submit.rs:161-185`), sender passes `--submit=claude`; asserts the delivered bytes equal the raw text with no chord appended. Red mutation: fall back to the caller's chord when the derive yields `None`.
3. **A mixed-agent group broadcast is per-session correct.** One `tab_group` holding a claude session, a codex session and a shell session, one enqueue with a single `submit`; asserts three different delivered byte strings. Red mutation: derive once from the first matched session and reuse that spec for the rest, which is exactly today's shape.
4. **`CHAN_AGENT` still wins over the command.** Session spawned with command `bash` and `CHAN_AGENT=codex` receives the codex chord; session spawned with command `codex` and `CHAN_AGENT=claude` receives the claude chord. Red mutation: derive from the command only, ignoring the spawn env. This mirrors the existing frame-side coverage at `crates/chan-server/src/routes/terminal.rs:1517-1534`.
5. **`cs terminal list --json` exposes the agent.** A `control_socket` test alongside `term_list_reports_the_pending_queue_depth` (`crates/chan-server/src/control_socket.rs:4945`) asserts `groups.<g>[0].agent == "codex"` for a codex-spawned session and null for a shell session. Red mutation: omit the field, or hardcode it to `"claude"`.
6. **Divergence is reported, if the item chooses to report it.** A test asserts the control reply string for a sender/target mismatch names both the requested and the applied agent. Red mutation: drop the divergence clause from the reply and return the bare `queued at position N`.
7. **Rich Prompt is untouched.** `crates/chan-server/src/routes/terminal.rs:1428-1450`, `crates/chan-server/src/routes/terminal.rs:1504-1561` and `web/packages/workspace-app/src/components/richPromptTerminalWiring.test.ts` pass unmodified. Red mutation: change `ServerFrame::Session`'s `submit_agent` field name or its derive inputs; both pins fail.
8. **`CHAN_MODE` has no readers.** A grep-shaped assertion in the docs check, or at minimum a recorded grep in the round journal, showing zero occurrences outside the sentence that warns about it. Red mutation: add a `std::env::var("CHAN_MODE")` read anywhere in `crates/`.
9. **Gate.** `cargo test -p chan-shell -p chan-library -p chan-server`, `cargo clippy --all-targets`, `cargo fmt --check` after the last edit, and `npm run check` plus the full vitest run in `web/` if any SPA-adjacent pin moves.
10. **Real-hardware proof, owner only.** A live four-tab team with one claude, one codex, one opencode and one gemini session, each poking every other with `--submit=<its own name>`, then with `--submit` omitted, then via a single `--tab-group` broadcast; every agent tab must show the message submitted, not parked. This cannot be proven in CI or on the sandbox host: it needs the four real agent CLIs attached to real PTYs, and the failure mode is visual (text sitting in a compose box). The owner runs it and records the twelve directed pairs plus the broadcast by result.

## Boundaries

In scope: the enqueue path in `crates/chan-library/src/terminal_sessions.rs`, `term_write` and `term_list` in `crates/chan-server/src/control_socket.rs`, `TerminalSessionSummary`, the `--submit` help and resolution point in `crates/chan-shell/src/cli.rs` and `crates/chan-shell/src/help.rs`, the corresponding sections of `crates/chan-shell/design.md`, and the `CHAN_AGENT` / `CHAN_MODE` wording in the team bootstrap generator (`crates/chan-server/src/routes/team_config.rs`).

Out of scope: the chord templates themselves (`crates/chan-shell/src/submit.rs:195-202`) and the `derive` rule (`crates/chan-shell/src/submit.rs:161-185`), both of which are correct and stay as they are; the Rich Prompt path and `ServerFrame::Session`; `submitMode.ts` and every SPA surface; the write-queue drainer's idle gating, batching and caps; and any change to the "no newline is appended" contract for raw writes.

Anchor corrections against the drafting brief: `TermWrite` on the wire is `crates/chan-shell/src/wire.rs:150-162` (the `submit` field at 160-161), not line 157; `SubmitAgent::derive` for the SPA frame is `crates/chan-server/src/routes/terminal.rs:946-951` inside `session_frame`, not 941; the bootstrap poke instructions are `crates/chan-server/src/routes/team_config.rs:402-414`, with the roster table itself at 663-729; and the trailing newline is trimmed at six sites, not five, the two extra being the notification batch framer at `crates/chan-library/src/terminal_sessions.rs:2150` and `crates/chan-library/src/terminal_sessions.rs:2201`. Every other anchor in the brief verified as given.
