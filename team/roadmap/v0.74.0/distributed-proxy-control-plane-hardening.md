# chan-gateway v0.74.0 control plane hardening proposal

Status: proposal, not yet accepted scope. This is the output of an adversarial security review of [distributed-proxy-control-plane.md](distributed-proxy-control-plane.md): three independent reviewers covered the proxy-to-controller protocol, the identity and origin chain to Chan Desktop, and the data plane with operational posture. Every critical and high claim was then re-verified against the code by the integrator before adoption; medium and low claims carry the reviewer's own citations. Branch citations name `v073/ctl` (read via `git show`; steps 1 to 4 are implemented there), unqualified paths are `main`. Steps 5 to 9 are unstarted, so findings touching consumers rest on the design text plus the pre-existing code they name. Grounded on 2026-07-20.

The review's thesis in one sentence: the control protocol authenticates *a fleet* but authorizes *individual proxies, admissions, ownership transfers, and kill confirmations on self-assertion*. One shared cleartext bearer plus self-asserted identity, rows, and command results is the root weakness. Three amendments (identity-issued admission tickets, per-proxy credentials with anti-displacement, numeric bounds with kill tombstones) collapse most of it, and none of them disturbs a stated non-goal.

## Threat model

- (a) external attacker able to reach a public listener;
- (b) authenticated tenant holding a valid PAT;
- (c) holder of the shared `DEVSERVER_PROXY_TOKEN`, obtainable from any one proxy node, an env-file leak, or the cleartext control path;
- (d) one fully compromised proxy node, which today also yields `IDENTITY_INTERNAL_TOKEN` and `DEVSERVER_GATE_SECRET`;
- (e) a compromised controller.

The design's trust section (distributed-proxy-control-plane.md:162-171) presents proxies as credential-poor nodes; they in fact hold three fleet-significant secrets, so (d) is a realistic capability and several findings are graded on it. Each finding names the weakest sufficient capability. No tests were executed; this was a read-only audit, and the mechanisms named were verified directly in source.

## Findings

### Control-plane trust

**1. Fabricated admissions seize any user's devserver and make the controller kill their live tunnel (critical; (c) suffices).** `AdmissionRequest` carries only `user` and `devserver_id` (distributed-proxy-control-plane.md:266), and `request_admission` demands no proof that a PAT was ever validated (`v073/ctl:gateway/crates/devserver-control/src/state.rs:373-452`); a key that is already live is count-neutral as a "reconnect" (rule 3, design line 313). `tunnel_up` then requires only a matching pending claim from the *same* session before installing the row as authoritative and issuing a kill for the victim's real registration UUID to their honest proxy (`state.rs:284-345`), which executes it (`v073/ctl:gateway/crates/devserver-proxy/src/control.rs:559-583`). A token holder joins as any unused id, publishes an empty snapshot, sends `AdmissionRequest` plus `TunnelUp` for `(alice, alice's devserver id)` (the id is not secret: it appears in every grantee's roster, `gateway/crates/identity/src/desktop_roster.rs:51`), and the controller itself evicts alice's live tunnel. Identity then mints entries pointing at the attacker's node, and her desktop's move-node lifecycle grants native authority to it. Her client's automatic reconnect re-enters the same rule-5 battle, so this is a permanent targeted kill loop, not a one-shot. A variant needs no kill at all: fabricated pending claims with fresh garbage devserver ids under the victim's public username fill her `MAX_DEVSERVERS_PER_USER` quota (rule 3's count-neutrality exempts only keys already present), so every genuine new registration she attempts is refused `at_capacity`, silently and indefinitely. The accepted live-first joining ruling does not cover this path: it governs joining snapshots, and this attack needs no snapshot at all. Amendment A1.

**2. One shared bearer is both fleet authentication and fleet authorization; the origin-template check is tautological for node naming (high; (c)).** The bearer is checked at the HTTP layer before `ClientHello` is read, so per-id credentials are not even expressible without reordering (`v073/ctl:gateway/crates/devserver-control/src/server.rs:174-196`). `validate_origin` expands the template with the attacker-chosen id and compares against the attacker-chosen claim, so both sides of the equality derive from the same `ClientHello`; the check blocks only off-template claims, never id choice (`server.rs:276-289`). `ProxyId::parse` accepts any one lowercase DNS label and no provisioned-membership test exists anywhere (`v073/ctl:gateway/crates/devserver-control-proto/src/lib.rs`; `state.rs:180`). The token is identical on every proxy node, and rotating it requires a simultaneous controller plus all-proxy restart. Chained with a stale per-node wildcard DNS record for a decommissioned node (design line 156 requires one record per node; line 530 makes provisioning an explicit non-goal), a squatted id plus an ACME certificate puts victims' browsers and desktop windows on attacker infrastructure with minted desktop authority. Amendment A2.

**3. Duplicate-id session replacement erases the incumbent's aggregate rows and can 503 the fleet (high; (c)).** On a duplicate id, `begin_session` logs a boot-id mismatch with `tracing::error!` and nothing more, then removes the incumbent session, which retain-drops every tunnel row and pending claim *before* the replacement publishes anything (`state.rs:180-229`, `remove_session` at `state.rs:1090-1116`). If the displaced session was the last `Active` one, readiness drops and the entire tunnel map is cleared (`state.rs:1118-1128`). The attacker flaps one connection per cycle; the real proxy's reconnect republishes; the attacker re-replaces. Sustained oscillation poisons the aggregate view, flaps identity rosters, and alternates the profile sweeper between skip and poisoned-but-complete. The accepted joining ruling cannot help here: the "immutable live winners" it depends on are deleted before any reconciliation runs. Amendment A3.

**4. Snapshot rows need no liveness proof; restart reconciliation elects gameable winners (high; (c), plus a controller-restart window for the second half).** `accept_snapshot` checks only the row count and intra-snapshot duplicate ids; staged rows are accepted with no per-row admission or liveness check (`state.rs:231-263`). A fabricated row for an offline user wins uncontested: identity reports her online and mints entries to the attacker's node. For a live user, initial restart reconciliation resolves duplicates by lexicographic `(proxy_id, registration_id)`, both attacker-chosen (`state.rs:855-910`, design line 320). On today's branch the same lexicographic rule is also applied to routine live joins (`reconcile_joining`, `state.rs:822-910`); the accepted ruling at design lines 37-46 covers exactly that case, the review independently confirms it is security load-bearing, and it must land first as scheduled. Amendment A4 covers the residual restart window the ruling deliberately leaves.

**5. `CommandResult` is believed unconditionally: a lying proxy defeats revocation silently (medium; (d)).** `command_result` validates the report's shape and then removes the registration on the proxy's say-so; nothing cross-checks the report against later state (`state.rs:509-559`). A compromised proxy reports `killed` and keeps serving, so PAT-revoke, block, and delete fan-outs (design lines 387-397) appear to succeed while the tunnel and its minted 24-hour gate cookies live on. The same lie lets poisoned rows pass the initial and joining reconciliation barriers, which wait on "successful command results" (design line 320). Amendment A8.

### Controller robustness

**6. Resource bounds are multiplicative; the delta path is uncapped; the actor re-publishes the world after every command and every second (high; (c) or (d)).** Snapshot rows are capped (100k, `devserver-control-proto/src/lib.rs:16`) but bytes are not: `TunnelRow.user` and `devserver_id` are unbounded `String`s (`lib.rs:121-127`), so a protocol-compliant snapshot can stage on the order of 780 MiB per session before `SnapshotEnd`. The runtime delta path has no per-session row cap at all (`tunnel_up`, `state.rs:284-345`), and pending claims are count-unbounded within their 15-second TTL. Every admission runs `distinct_for_user`, a full scan of all tunnel and pending keys into a `HashSet` (`state.rs:1141-1149`), and the actor clones, sorts, and full-compares the entire tunnel and proxy view after every command and every 1-second tick (`v073/ctl:gateway/crates/devserver-control/src/actor.rs`, `publish_watches`). One malicious session dripping small frames wedges the single actor task, heartbeats slip the 15-second dead window, honest sessions are retired fleet-wide, and the component the design makes a hard single point (design line 110) falls over. Amendment A5.

**7. A version-mismatch `Shutdown` exits the whole proxy process; every lockstep upgrade is a guaranteed fleet-wide tunnel kill (medium; operational, or (e) on demand).** A hello rejection sends `Shutdown` (`server.rs:240-245`); the proxy maps `Shutdown` to `AttemptError::Permanent` (`control.rs:601`), `supervise` bails on `Permanent` (`control.rs:263`), and the process exits (`main.rs` first-exit handling), instantly killing all of its live tunnels with none of the 30-second grace a controller loss grants (design line 110). In either upgrade order, every not-yet-upgraded node dies this way. The exact-version pin itself provides no security property: the expected version string is public and trivially claimed by an impostor. It is config-fault detection, and its current price is a thundering herd. Amendment A9.

### Secrets, transport, and revocation

**8. `DEVSERVER_GATE_SECRET` on every proxy: one compromised node mints owner-level cookies for any user it ever saw, valid fleet-wide (high; (d)).** Entry and session tokens share one HS256 secret, with `sub`, `role`, `drv`, and `aud` all minter-chosen and `iss` "verified opportunistically" (`gateway/crates/gateway-common/src/devserver_gate.rs:154-209`); the proxy verify path binds only signature, `aud` equal to the inbound host, and `drv` equal to a live local registration. `drv` is the stable SHA-256 of the user's PAT, known to any node that ever validated it. A compromised p2 therefore mints a 24-hour `typ:"session"` cookie with `role:"owner"` for alice's devserver and presents it at whichever node currently hosts her, passing as owner. PAT revocation of the attacker is irrelevant because the `sub` is forged, and the minting capability survives p2's reimaging until a fleet-wide secret rotation that the code itself notes invalidates every live token (`devserver_gate.rs:24-26`). Amendment A6.

**9. Raw PATs and every internal bearer cross the wire cleartext; `/internal/v1/*` rides identity's public application (high; (a) the moment the overlay assumption fails).** The proxy posts the raw user PAT as JSON to identity over plain HTTP on every tunnel handshake (`gateway/crates/devserver-proxy/src/identity_validator.rs:79-86`, default `http://127.0.0.1:7000`); `/internal/v1/tokens/validate` is a sub-router merged into identity's single public axum application behind one route layer (`gateway/crates/identity/src/http.rs:145-148`), reachable wherever identity is; the control channel is h2c (design line 232); the container images default every service to `BIND_ADDR=0.0.0.0` (`packaging/docker/gateway.Dockerfile`). The only mitigation named for all of this is design line 164's "private deployment network such as WireGuard". A passive observer on a mis-provisioned or co-tenant network collects the proxy token, the admin token, and raw user PATs, which are full-impersonation credentials. Amendment A7.

**10. Failed revoke, block, and delete kills leave tunnels and 24-hour cookies serving; the design's named backstop does not fire (medium; reliability gap, silent under (d)).** The call sites are best-effort log-and-continue (`gateway/crates/identity/src/http.rs:969-988` for revoke, `:738-762` for account delete, `gateway/crates/profile/src/http.rs:730-745` for block), and design line 397 retains those semantics while naming the gap-closer as "PAT validation plus the proxy's bounded control-loss behavior". Neither fires here: there is no next PAT validation for an established tunnel, and the 30-second control-loss grace triggers on control disconnect, not on a failed or partial kill over a healthy session, whose 502 the caller merely logs (design line 381). Amendment A10.

### Browser, tenant, and desktop surface

**11. Cross-tenant cookie tossing: `chan.app` is not on the Public Suffix List and no cookie carries a `__Host-` prefix (medium; (b)).** Verified against the live PSL on 2026-07-20: zero `chan.app` entries, so the eTLD+1 is `chan.app` and JavaScript served from one tenant's devserver origin (any authenticated user can serve arbitrary JS from theirs) may set cookies with `Domain=p1.usr.chan.app`, `Domain=usr.chan.app`, or `Domain=chan.app`. Every gateway cookie is host-only today (no `Domain=` attribute anywhere under `gateway/`), but host-only scope does not stop a parent-domain cookie of the same name from being sent alongside: shadow `devserver_gate` or `devserver_csrf` cookies yield a cross-tenant logout/404 denial of service with attacker-chosen `Max-Age`, and a shadow `id_session` is a session-fixation candidate against identity. Riding the victim's session fails (the per-request `aud` and `drv` binding holds), so this is DoS plus the fixation candidate; the server-side duplicate-cookie resolution order that fixation depends on is unverified, and the amendment closes the class regardless. Amendment A11.

**12. Stolen-PAT cross-node seizure: singleton semantics preserved, blast radius amplified, no flap damping (medium; (b) with a stolen PAT).** Last-writer-wins replacement is today's documented registry semantics (`crates/chan-tunnel-server/src/registry.rs:9-15`), and rules 3 and 5 extend it across nodes: the thief dials any proxy, is admitted count-neutral, and the controller kills the victim's registration on its home node. Unlike the singleton world, identity now re-mints all future entry origins and the desktop roster onto the thief's node, including the move-node lifecycle that grants native IPC to the new origin. The victim's client auto-reconnects and the kill battle flaps indefinitely; nothing in the state machine damps repeated ownership transfers of one key. Amendment A12.

**13. A compromised controller escapes the namespace on the browser leg; identity delegates the node-base check (medium; (e)).** The template validation behind "validated node base" lives inside the controller (`server.rs:276-289`), so a compromised controller skips it: a row carrying `proxy_base_url: https://evil.example` yields `aud` and `entry_url` on `alice--disc.evil.example`, and share landings 303 browsers straight there under a validly signed entry JWT. Replay of that JWT at real proxies fails the `aud` check, so this is phishing and token capture rather than session theft; the desktop rejects out-of-apex origins, so desktop authority stays contained. Design line 160's "identity must never construct or accept an arbitrary node URL" is currently enforced by the party the rule distrusts. Amendment A13.

**14. Desktop move-node: the pin reset and the tenant-label binding are unspecified (low; confusion and bug assist, (e) for the worst case).** Verified on main: Tauri authority is minted only after `validate_gateway_entry` (`desktop/src-tauri/src/main.rs:2490`), and roster handling never mints. Two gaps remain for step 7: a legitimate node move trips the pinned-origin check (`desktop/src-tauri/src/devserver.rs:722-728`) unless the row lifecycle explicitly clears the pin and re-enters validation, and nothing binds the origin's first label to the entry response's owner and devserver id, so a row-mixing bug or a compromised controller can cross one user's fields with another's tenant label unnoticed. Amendment A14. Found in passing, not a design finding: `is_loopback_gateway_host` treats any DNS name beginning `127.` (for example `127.example.com`) as loopback and waives HTTPS (`devserver.rs:374-384`); it should parse IP literals.

### Observability and operations

**15. The audit trail cannot distinguish a hijack from a reconnect (medium; detection gap under (c) or (d)).** Design line 354's log list omits ownership change. `state.rs` contains exactly four `tracing` sites (boot-id mismatch, command timeout, heartbeat expiry, a ping debug); the `tunnel_up` kill path, every admission decision, and every `issue_kill` emit nothing, and admin kills carry no initiator attribution behind one shared token. After finding 1 or 2 is exercised, controller logs show at most routine heartbeats. Amendment A15.

**16. The aggregate admin API concentrates fleet PII and topology under one fan-out token (low).** `TunnelView.peer_addr` is each devserver user's client IP address, aggregated fleet-wide in one response; `ProxyView` adds package versions and boot ids, useful targeting intelligence. `DEVSERVER_ADMIN_TOKEN` is one value shared by identity, profile, and the operator CLI (`packaging/gateway/scripts/configure.sh:100-106`), two of which are internet-facing services, and the SSE watch endpoints have no connection bound. Amendment A16.

**17. Version agreement is enforced only on the proxy-to-control handshake (low; operational).** Identity, profile, and the admin CLI reach the controller over the plain admin HTTP tree with no version exchange; only release-asset validation (design line 431) guards that skew. Amendment A17.

## Desired contract

Amend the design as follows before step 5 begins. Each amendment names the design section it changes. A1, A2, and A3 are load-bearing; most other amendments assume them.

**A1 (Synchronous fleet admission): an `AdmissionRequest` must be cryptographic evidence of a validated PAT.** Identity's `/internal/v1/tokens/validate` response gains a short-lived admission ticket, `HMAC(fleet_admission_secret, user | devserver_id | registration_id | proxy_id | expiry)`, minted during the PAT validation the proxy already performs before calling admission (design line 304; the validate request gains `registration_id` and `proxy_id` fields). The proxy relays the ticket verbatim on `AdmissionRequest`; the controller verifies the MAC, the expiry, and that the ticket's `proxy_id` equals the session's before creating a pending claim. One new identity-to-controller shared secret, config-only; the controller stays database-free and the protocol gains one field. Capability (c) can then fabricate no admission, and (d) is confined to admitting keys whose PATs were actually presented to that node; fabricated snapshot rows and session displacement are separate holes, covered by A3 and A4. Finding 1's cap-poisoning variant and most claim-flooding die with the same change, because both run through `AdmissionRequest`. A cheaper interim (the proxy MACs with `DEVSERVER_GATE_SECRET`) does not bound (d), since a compromised proxy holds that secret, and is not recommended as the endpoint.

**A2 (Trust and network boundaries / Control transport): per-proxy credentials.** Check the bearer after `ClientHello` against the claimed id, or bind the id into the connect path. Preferred shape: derived tokens, `expected(p) = HMAC(proxy_master_secret, p)`, verified constant-time. No membership state is introduced, so the acceptance criterion at design line 516 (no static membership edit) survives; provisioning a node is deriving a value offline. The acceptable alternative is a provisioned `{proxy_id: token}` map, which buys an id allowlist and per-node revocation at the price of a config edit per node; the ADR should own this choice. Either way, a stolen token's blast radius shrinks from "any node name in the namespace" to "that one node". Document a two-value overlap window so rotation needs no fleet-wide admission outage, and add the DNS hygiene rule to the public host contract: never publish `*.{id}.` records for unprovisioned ids, and delete them at decommission (finding 2's stale-record chain).

**A3 (Controller state and readiness): anti-displacement and deferred retirement, hard-dependent on A9.** Refuse to displace a heart-healthy incumbent: a duplicate connect for an `Active`, ping-responsive id is rejected instead of honored, and legitimate restarts rejoin after TCP death is detected (clean close immediately, heartbeat within 15 seconds, both already implemented). Until A9 lands, the rejection must not be a `Shutdown`, because `Shutdown` exits the proxy process today (finding 7) and the race is routine: both sides run the same 15-second dead window, so a proxy redialing after a control-path stall would be told to die with all of its tunnels. Use a distinct retryable close frame, or sequence A9 first. When replacement does proceed, keep the incumbent's rows published, marked superseded, until the replacement's snapshot completes or its 30-second deadline fires, then retire the loser. Sessions are stored one per proxy id today, so holding an incumbent alongside a joining replacement is a structural change to session storage, not a staging tweak; the implementation round sizes it. One residual is accepted and named: the predicate protects only `Active` incumbents, so a per-id replacement hold-down plus operator alert covers repeated displacement of a `Joining` session. After A3, a boot-id mismatch is unambiguous attack or fault signal: alert on it, do not just log it.

**A4 (the accepted ruling plus restart reconciliation): land the ruling first; fail closed on cross-proxy duplicates at restart.** The ruling at design lines 37-46 is confirmed security load-bearing and stays the first code of the next round. Residual hardening it does not cover: (i) initial restart reconciliation kills *all* copies of a cross-proxy duplicate key and lets the real client reconnect, instead of electing a lexicographic winner whose sort keys are both attacker-chosen; duplicates across nodes after a restart mean a kill was lost with the dead controller, and a spurious reconnect beats a hijacked origin; (ii) within one controller lifetime, a non-empty snapshot presented under a boot id that differs from the boot id this controller last saw for that proxy id is quarantined and alerted: a fresh proxy process has an empty registry, so rows under a changed boot id are fabrication or corruption. This needs a small bounded proxy-id-to-boot-id memory that survives session removal within the process lifetime, and it is exempt during initial convergence, where every boot id is unseen by construction and accepting the fleet's grace-period snapshots is the whole point (design line 320); (iii) reconciliation loser kills, especially capacity trims, get a distinct warn-level line that operators alert on, and the ADR states plainly that restart-time reconciliation trusts first-presented rows by construction.

**A5 (Control transport / Controller state and readiness): numeric bounds.** Bound cumulative snapshot bytes (for example 16 MiB) in addition to rows; apply the 100k per-session cap to live rows inclusive of deltas; cap pending claims per session and globally; coalesce watch publication to the 1-second tick behind a dirty flag instead of after every command. All constants-and-guards changes inside the existing actor and adapter shape.

**A6 (Trust and network boundaries): per-node gate keys.** Derive `node_secret = HMAC(gate_master, proxy_id)`; identity already learns `proxy_id` from the aggregate row (design line 375) and derives that node's key at entry mint; each proxy holds only its own derived secret. Still symmetric HS256, no cookie-shape change, and master rotation still rotates everything, but capability (d) degrades to the node's inherent data-path power (it can already inject gateway-assertion headers for its own tunnels) instead of fleet-wide cookie minting.

**A7 (Trust and network boundaries / Packaging): make the overlay load-bearing on paper and smaller in fact.** The ADR states that WireGuard or equivalent provides confidentiality for raw PATs and internal bearers, not merely reachability. Accept `https` `DEVSERVER_CONTROL_URL` and `IDENTITY_URL` with operator-supplied certificates for deployments that want defense in depth (the per-node wildcard certificates already set the provisioning precedent). Split identity's `/internal/v1/*` tree onto a second listener bound to the WireGuard interface address so the internal bearer is never internet-reachable, and bind internal listeners to the provisioned interface address instead of `0.0.0.0` in the packaged env files. Document rotation order for both tokens. Carried unverified: the repository ships no edge or nginx configuration, so today's `/internal` exposure rests on the merged router plus whatever edge the deployment runs.

**A8 (Controller state and readiness): kill tombstones.** Keep a bounded memory-only FIFO (for example 10k entries, expiry aligned to the 24-hour gate-cookie lifetime) of registration UUIDs whose kills were confirmed, per proxy id. A later snapshot row or `TunnelUp` carrying a tombstoned UUID proves the proxy lied: force resync, refuse the session's snapshot, and page an operator. This converts silent defiance into a loud fail-closed event without durable state.

**A9 (Proxy failure semantics): do not die on `Shutdown`.** A hello rejection or `Shutdown` during connect means retry with long capped backoff while staying unready: no new admissions, existing tunnels keep serving, operator alerted. Reserve process exit for local configuration faults. This is one branch in `supervise`, and it removes both the guaranteed fleet-wide tunnel kill on every lockstep upgrade and the (e)-triggered mass exit, at the price of stale binaries lingering unready during a botched rollout, which is the safer failure direction.

**A10 (Identity, profile, and admin changes): bounded revocation retry.** On `Err` or a partial-kill 502 from the controller, identity and profile re-issue the user kill with exponential backoff over roughly five minutes and record final failure in `auth_audit`, which both services already write. The ADR states plainly that the control kill is the only revocation mechanism for an established tunnel and its minted cookies.

**A11 (Public host and routing contract): `__Host-` cookie names at the cutover.** Rename `devserver_gate`, `devserver_csrf`, and `id_session` to their `__Host-` forms; all three already satisfy the set-rules (`Secure`, `Path=/`, no `Domain`), so browsers then reject parent-domain shadows at set time. File a PSL submission for `usr.chan.app` as a long-lead operational follow-up. v0.74.0 is the exact-version cutover, so the rename rides it cleanly.

**A12 (Synchronous fleet admission): ownership-transfer damping.** Reject or quarantine a key whose authoritative owner changed within the last N seconds, a small actor rule on the `Instant`s the controller already holds; it is orthogonal to the live-first joining ruling, which governs a different path. Document PAT rotation as the remediation for theft.

**A13 (Identity, profile, and admin changes): identity re-anchors node bases.** Before minting, identity enforces that the row's node base host is exactly one DNS label under `DEVSERVER_PROXY_ORIGIN` with matching scheme and effective port, the same string check the desktop performs, so design line 160's rule becomes self-enforcing instead of delegated to the controller.

**A14 (Chan Desktop discovery and exact-origin validation): bind the label, name the pin.** The entry validator requires the origin's first label to equal `{username}--{devserver_id[..12]}` lowercased, and the move-node lifecycle explicitly clears the per-row pinned origin and re-enters through `validate_gateway_entry` with `pinned_origin = None`. Fix `is_loopback_gateway_host` to parse IP literals instead of prefix-matching `127.`.

**A15 (Proxy failure semantics / logging): make hijacks audible.** Extend design line 354's list: every ownership change at WARN with `user`, `devserver_id`, old and new `proxy_id` and `boot_id`, and the control peer address; every admission denial and every cross-session `Stale`; every admin kill and read with caller peer address. Document the flap pattern, one key changing nodes within seconds, as the hijack signature operators alert on.

**A16 (Aggregate admin API): PII and watcher bounds.** Document `peer_addr` as PII in the admin documentation, keep it on per-user detail reads rather than the fleet list, and bound concurrent SSE watchers per token.

**A17 (Aggregate admin API): version header.** `/admin/v1/*` responses carry the controller's `package_version`; `DevserverControlClient` logs or refuses on mismatch, extending the immutable-version acceptance criterion (design line 523) to the consumer leg.

## Verified sound, kept out of the findings

The review tried and failed to break these: stale incarnations cannot mutate or disconnect a replacement session (`require_key`, with direct test coverage including a retired connection's late frames, disconnects, and command results); both bearers are compared constant-time; a proxy cannot register an off-template origin (finding 2 is about which on-template id, not arbitrary origins); every internal queue is bounded and closes the session on overflow rather than dropping state; the 30-second snapshot deadline is absolute across the whole phase; the 10-second admission wait, `AdmissionCancel`, and 15-second claim expiry are implemented as specified; kills target registration UUIDs, so delayed commands cannot hit successors; controller reads fail closed at 503 while unready; PAT guessing is throttled at both proxy and identity; gate cookies bind `aud` and `drv` per request, so cross-tenant cookie theft or transplant cannot ride a session; HS256 is hard-required with `alg:none` rejected and `typ` dispatched; `owner` and `disc` label validation forbids dots and lookalikes; packaging keeps secrets out of images and examples.

The accepted joining ruling and the step-4 barrier evidence were review inputs, not targets; the only addition to them here is A4's residual list.

## Boundaries

- Every amendment fits the design's existing shape: in-memory singleton controller, no database, no controller HA, no proxy-to-proxy path, no mixed-version support, no automatic DNS or certificate provisioning. The explicit non-goals at design lines 525-532 stand.
- This document proposes spec-level amendments only; it changes no code. Implementation lands in the step 5 to 9 rounds under the design's red-before-green discipline.
- The findings grade the design, not the `v073/ctl` implementation, which implements the design as written; most findings are properties of the design text.
- Carried unverified, honestly: the server-side duplicate-cookie resolution order behind the `id_session` fixation sub-claim (A11 closes the class regardless); whether any deployment edge blocks `/internal/*` (A7 closes it regardless); the deployment binding of the proxy h2c listener (step-9 work).

## Acceptance

- A1, A2, and A3 land before any step-5 code, with A3's duplicate-connect rejection shipped only alongside A9 or as a retryable close frame; A4's ruling lands first as already scheduled.
- Each amendment is reflected in the named design-doc section and in ADR `gateway/docs/adr/0002-control-plane-owns-proxy-fleet-state.md`, including the honesty items: restart reconciliation trusts first-presented rows by construction, the control kill is the only revocation for established tunnels, and the overlay is load-bearing for confidentiality.
- The design's Verification section gains adversarial cases mirroring the findings: a fabricated `AdmissionRequest` plus `TunnelUp` without a ticket is refused and the prior owner is never killed; a duplicate connect against a healthy incumbent is rejected; a non-empty snapshot under an unseen boot id is quarantined; a tombstoned UUID reappearing forces resync and refuses the snapshot; the new bounds are proven under paused Tokio time alongside the existing time-dependent tests.
- No amendment expands scope into the design's non-goals, and the version cutover stays exact.
