Status: **accepted, implemented, and green on `v074/ctl`; ready to merge when
the v0.74.0 integration window opens.** The findings below remain the pinned
2026-07-20 pre-hardening audit

## Implementation closure, 2026-07-21

The mandatory merge set is implemented:

- admission requires an identity-signed, refreshable Ed25519 lease bound to the
  immutable owner, devserver, registration, and provisioned proxy;
- proxy and admin credentials are allowlisted and scope-specific, with bounded
  rotation rings and no cross-scope reuse;
- duplicate-id displacement, restart rows, frames, strings, snapshots, pending
  claims, actor work, watchers, and fleet/user capacity are bounded and fail
  closed;
- browser handoff is a body-only, 30-second, single-use Ed25519 credential with
  no PII, followed by a bounded opaque proxy-local session with a one-hour
  absolute maximum;
- grant, block, PAT, and account denials reserve durable generation-fenced
  tunnel/session revocation work in the same database transaction;
- HTTP and WebSocket operations register revocation guards, enforce exact
  browser boundaries, and do not acknowledge revoke until matching transports
  have drained; a timeout remains tombstoned for honest retry;
- Desktop validates immutable roster and origin bindings, sends exact WebSocket
  Origin, and reuses/refreshes one opaque session without the old 16-session
  window churn;
- systemd and Kubernetes examples split service identities, Secrets, admin
  scopes, database roles, and migration ownership, and enforce explicit
  protected transport plus default-deny network policy; and
- the three-node E2E covers signed admission/entry, ownership isolation,
  shared access, revoke, controller restart/loss, placement movement, and the
  complete diagonal/off-node matrix.

### Honest residual boundary

A fully compromised proxy assigned a tunnel can observe the transient PAT at
validation time and can mint request assertions for tunnels assigned to that
node. Signed leases, node-scoped entry verification, scoped credentials, and
deployment isolation stop that capability from becoming another node's or the
whole fleet's authority; they do not make the assigned node a trusted execution
environment. Isolation, deprovisioning, and PAT rotation remain the incident
response. The singleton controller is also still an availability dependency;
HA and durable fleet state remain explicit non-goals.

## Historical audit and proposal

This section is the output of an adversarial security review of [distributed-proxy-control-plane.md](distributed-proxy-control-plane.md): three independent reviewers covered the proxy-to-controller protocol, the identity and origin chain to Chan Desktop, and the data plane with operational posture. A follow-up end-to-end tenant-isolation pass traced browser credentials through devserver-proxy, the per-request gateway assertion, chan-server route authorization, workspace and terminal tenants, grant revocation, and username changes. Every critical and high claim was then re-verified against the code before adoption; medium and low claims carry the reviewer's own citations. Branch citations name `v073/ctl` (read via `git show`; steps 1 to 4 are implemented there), unqualified paths are `main`. Steps 5 to 9 were unstarted at review time, so findings touching consumers rest on the design text plus the pre-existing code they name. Grounded on 2026-07-20.

The review's thesis in one sentence: the control protocol authenticates *a fleet* but authorizes *individual proxies, admissions, ownership transfers, and kill confirmations on self-assertion*, while the data path authenticates a browser session without consistently binding immutable ownership, the exact binary grant, revocation state, or browser origin. Shared cleartext bearers plus self-asserted control rows are the control-plane root weakness; mutable username ownership, an ambiguous sharing contract, stateless 24-hour sessions, and unchecked WebSocket origins are the user-data root weaknesses. A1 through A4 and A18 through A21 are load-bearing before broad consumer migration.

## Threat model

- (a) external attacker able to reach a public listener;
- (b) authenticated tenant holding a valid PAT;
- (c) holder of the shared `DEVSERVER_PROXY_TOKEN`, obtainable from any one proxy node, an env-file leak, or the cleartext control path;
- (d) one fully compromised proxy node, which today also yields `IDENTITY_INTERNAL_TOKEN` and `DEVSERVER_GATE_SECRET`;
- (e) a compromised controller;
- (f) an accepted grantee whose access must remain confined to the one explicitly shared devserver and end when that grant is revoked;
- (g) malicious JavaScript on one tenant origin, visited in a browser that also holds a valid gate cookie for another tenant origin.

The design's trust section (distributed-proxy-control-plane.md:162-171) presents proxies as credential-poor nodes; they in fact hold three fleet-significant secrets, so (d) is a realistic capability and several findings are graded on it. Each finding names the weakest sufficient capability. No runtime tests were executed; the source audit itself was read-only, and the mechanisms named were verified directly in source.

## Findings

### Control-plane trust

**1. Fabricated admissions seize any user's devserver and make the controller kill their live tunnel (critical; (c) suffices).** `AdmissionRequest` carries only `user` and `devserver_id` (distributed-proxy-control-plane.md:266), and `request_admission` demands no proof that a PAT was ever validated (`v073/ctl:gateway/crates/devserver-control/src/state.rs:373-452`); a key that is already live is count-neutral as a "reconnect" (rule 3, design line 313). `tunnel_up` then requires only a matching pending claim from the *same* session before installing the row as authoritative and issuing a kill for the victim's real registration UUID to their honest proxy (`state.rs:284-345`), which executes it (`v073/ctl:gateway/crates/devserver-proxy/src/control.rs:559-583`). A token holder joins as any unused id, publishes an empty snapshot, sends `AdmissionRequest` plus `TunnelUp` for `(alice, alice's devserver id)` (the id is not secret: it appears in every grantee's roster, `gateway/crates/identity/src/desktop_roster.rs:51`), and the controller itself evicts alice's live tunnel. Identity then mints entries pointing at the attacker's node, and her desktop's move-node lifecycle grants native authority to it. Her client's automatic reconnect re-enters the same rule-5 battle, so this is a permanent targeted kill loop, not a one-shot. A variant needs no kill at all: fabricated pending claims with fresh garbage devserver ids under the victim's public username fill her `MAX_DEVSERVERS_PER_USER` quota (rule 3's count-neutrality exempts only keys already present), so every genuine new registration she attempts is refused `at_capacity`, silently and indefinitely. The accepted live-first joining ruling does not cover this path: it governs joining snapshots, and this attack needs no snapshot at all. Amendment A1.

**2. One shared bearer is both fleet authentication and fleet authorization; the origin-template check is tautological for node naming (high; (c)).** The bearer is checked at the HTTP layer before `ClientHello` is read, so per-id credentials are not even expressible without reordering (`v073/ctl:gateway/crates/devserver-control/src/server.rs:174-196`). `validate_origin` expands the template with the attacker-chosen id and compares against the attacker-chosen claim, so both sides of the equality derive from the same `ClientHello`; the check blocks only off-template claims, never id choice (`server.rs:276-289`). `ProxyId::parse` accepts any one lowercase DNS label and no provisioned-membership test exists anywhere (`v073/ctl:gateway/crates/devserver-control-proto/src/lib.rs`; `state.rs:180`). The token is identical on every proxy node, and rotating it requires a simultaneous controller plus all-proxy restart. Chained with a stale per-node wildcard DNS record for a decommissioned node (design line 156 requires one record per node; line 530 makes provisioning an explicit non-goal), a squatted id plus an ACME certificate puts victims' browsers and desktop windows on attacker infrastructure with minted desktop authority. Amendment A2.

**3. Duplicate-id session replacement erases the incumbent's aggregate rows and can 503 the fleet (high; (c)).** On a duplicate id, `begin_session` logs a boot-id mismatch with `tracing::error!` and nothing more, then removes the incumbent session, which retain-drops every tunnel row and pending claim *before* the replacement publishes anything (`state.rs:180-229`, `remove_session` at `state.rs:1090-1116`). If the displaced session was the last `Active` one, readiness drops and the entire tunnel map is cleared (`state.rs:1118-1128`). The attacker flaps one connection per cycle; the real proxy's reconnect republishes; the attacker re-replaces. Sustained oscillation poisons the aggregate view, flaps identity rosters, and alternates the profile sweeper between skip and poisoned-but-complete. The accepted joining ruling cannot help here: the "immutable live winners" it depends on are deleted before any reconciliation runs. Amendment A3.

**4. Snapshot rows need no liveness proof; restart reconciliation elects gameable winners (high; (c), plus a controller-restart window for the second half).** `accept_snapshot` checks only the row count and intra-snapshot duplicate ids; staged rows are accepted with no per-row admission or liveness check (`state.rs:231-263`). A fabricated row for an offline user wins uncontested: identity reports her online and mints entries to the attacker's node. For a live user, initial restart reconciliation resolves duplicates by lexicographic `(proxy_id, registration_id)`, both attacker-chosen (`state.rs:855-910`, design line 320). On today's branch the same lexicographic rule is also applied to routine live joins (`reconcile_joining`, `state.rs:822-910`); the accepted ruling at design lines 37-46 covers exactly that case, the review independently confirms it is security load-bearing, and it must land first as scheduled. Amendment A4 covers the residual restart window the ruling deliberately leaves.

**5. `CommandResult` is believed unconditionally: a lying proxy defeats revocation silently (medium; (d)).** `command_result` validates the report's shape and then removes the registration on the proxy's say-so; nothing cross-checks the report against later state (`state.rs:509-559`). A compromised proxy reports `killed` and keeps serving, so PAT-revoke, block, and delete fan-outs (design lines 387-397) appear to succeed while the tunnel and its minted 24-hour gate cookies live on. The same lie lets poisoned rows pass the initial and joining reconciliation barriers, which wait on "successful command results" (design line 320). Amendment A8.

### Controller robustness

**6. Resource bounds are multiplicative; the delta path is uncapped; the actor re-publishes the world after every command and every second (high; (c) or (d)).** Snapshot rows are capped (100k, `devserver-control-proto/src/lib.rs:16`) but bytes are not: `TunnelRow.user` and `devserver_id` are unbounded `String`s (`lib.rs:121-127`), so a protocol-compliant snapshot can stage on the order of 780 MiB per session before `SnapshotEnd`. The runtime delta path has no per-session row cap at all (`tunnel_up`, `state.rs:284-345`), and pending claims are count-unbounded within their 15-second TTL. Every admission runs `distinct_for_user`, a full scan of all tunnel and pending keys into a `HashSet` (`state.rs:1141-1149`), and the actor clones, sorts, and full-compares the entire tunnel and proxy view after every command and every 1-second tick (`v073/ctl:gateway/crates/devserver-control/src/actor.rs`, `publish_watches`). One malicious session dripping small frames wedges the single actor task, heartbeats slip the 15-second dead window, honest sessions are retired fleet-wide, and the component the design makes a hard single point (design line 110) falls over. Amendment A5.

**7. A version-mismatch `Shutdown` exits the whole proxy process; every lockstep upgrade is a guaranteed fleet-wide tunnel kill (medium; operational, or (e) on demand).** A hello rejection sends `Shutdown` (`server.rs:240-245`); the proxy maps `Shutdown` to `AttemptError::Permanent` (`control.rs:601`), `supervise` bails on `Permanent` (`control.rs:263`), and the process exits (`main.rs` first-exit handling), instantly killing all of its live tunnels with none of the 30-second grace a controller loss grants (design line 110). In either upgrade order, every not-yet-upgraded node dies this way. The exact-version pin itself provides no security property: the expected version string is public and trivially claimed by an impostor. It is config-fault detection, and its current price is a thundering herd. Amendment A9.

### Secrets, transport, and revocation

**8. `DEVSERVER_GATE_SECRET` on every proxy: one compromised node mints owner-level cookies for any user it ever saw, valid fleet-wide (high; (d)).** Entry and session tokens share one HS256 secret, with `sub`, `role`, `drv`, and `aud` all minter-chosen and `iss` "verified opportunistically" (`gateway/crates/gateway-common/src/devserver_gate.rs:154-209`); the proxy verify path binds only signature, `aud` equal to the inbound host, and `drv` equal to a live local registration. `drv` is the stable SHA-256 of the user's PAT, known to any node that ever validated it. A compromised p2 therefore mints a 24-hour `typ:"session"` cookie with `role:"owner"` for alice's devserver and presents it at whichever node currently hosts her, passing as owner. PAT revocation of the attacker is irrelevant because the `sub` is forged, and the minting capability survives p2's reimaging until a fleet-wide secret rotation that the code itself notes invalidates every live token (`devserver_gate.rs:24-26`). Amendment A6.

**9. Raw PATs and every internal bearer cross the wire cleartext; `/internal/v1/*` rides identity's public application (high; (a) the moment the overlay assumption fails).** The proxy posts the raw user PAT as JSON to identity over plain HTTP on every tunnel handshake (`gateway/crates/devserver-proxy/src/identity_validator.rs:79-86`, default `http://127.0.0.1:7000`); `/internal/v1/tokens/validate` is a sub-router merged into identity's single public axum application behind one route layer (`gateway/crates/identity/src/http.rs:145-148`), reachable wherever identity is; the control channel is h2c (design line 232); the container images default every service to `BIND_ADDR=0.0.0.0` (`packaging/docker/gateway.Dockerfile`). The only mitigation named for all of this is design line 164's "private deployment network such as WireGuard". A passive observer on a mis-provisioned or co-tenant network collects the proxy token, the admin token, and raw user PATs, which are full-impersonation credentials. Amendment A7.

**10. Failed revoke, block, and delete kills leave tunnels and 24-hour cookies serving; the design's named backstop does not fire (medium; reliability gap, silent under (d)).** The call sites are best-effort log-and-continue (`gateway/crates/identity/src/http.rs:969-988` for revoke, `:738-762` for account delete, `gateway/crates/profile/src/http.rs:730-745` for block), and design line 397 retains those semantics while naming the gap-closer as "PAT validation plus the proxy's bounded control-loss behavior". Neither fires here: there is no next PAT validation for an established tunnel, and the 30-second control-loss grace triggers on control disconnect, not on a failed or partial kill over a healthy session, whose 502 the caller merely logs (design line 381). Amendment A10.

### Browser, tenant, and desktop surface

**11. Cross-tenant cookie tossing: `chan.app` is not on the Public Suffix List and no cookie carries a `__Host-` prefix (medium; (b)).** Verified against the live PSL on 2026-07-20: zero `chan.app` entries, so the eTLD+1 is `chan.app` and JavaScript served from one tenant's devserver origin (any authenticated user can serve arbitrary JS from theirs) may set cookies with `Domain=p1.usr.chan.app`, `Domain=usr.chan.app`, or `Domain=chan.app`. Every gateway cookie is host-only today (no `Domain=` attribute anywhere under `gateway/`), but host-only scope does not stop a parent-domain cookie of the same name from being sent alongside: shadow `devserver_gate` or `devserver_csrf` cookies yield a cross-tenant logout/404 denial of service with attacker-chosen `Max-Age`, and a shadow `id_session` is a session-fixation candidate against identity. Riding the victim's session fails (the per-request `aud` and `drv` binding holds), so this is DoS plus the fixation candidate; the server-side duplicate-cookie resolution order that fixation depends on is unverified, and the amendment closes the class regardless. Amendment A11.

**12. Stolen-PAT cross-node seizure: singleton semantics preserved, blast radius amplified, no flap damping (medium; (b) with a stolen PAT).** Last-writer-wins replacement is today's documented registry semantics (`crates/chan-tunnel-server/src/registry.rs:9-15`), and rules 3 and 5 extend it across nodes: the thief dials any proxy, is admitted count-neutral, and the controller kills the victim's registration on its home node. Unlike the singleton world, identity now re-mints all future entry origins and the desktop roster onto the thief's node, including the move-node lifecycle that grants native IPC to the new origin. The victim's client auto-reconnects and the kill battle flaps indefinitely; nothing in the state machine damps repeated ownership transfers of one key. Amendment A12.

**13. A compromised controller escapes the namespace on the browser leg; identity delegates the node-base check (medium; (e)).** The template validation behind "validated node base" lives inside the controller (`server.rs:276-289`), so a compromised controller skips it: a row carrying `proxy_base_url: https://evil.example` yields `aud` and `entry_url` on `alice--disc.evil.example`, and share landings 303 browsers straight there under a validly signed entry JWT. Replay of that JWT at real proxies fails the `aud` check, so this is phishing and token capture rather than session theft; the desktop rejects out-of-apex origins, so desktop authority stays contained. Design line 160's "identity must never construct or accept an arbitrary node URL" is currently enforced by the party the rule distrusts. Amendment A13.

**14. Desktop move-node: the pin reset and the tenant-label binding are unspecified (low; confusion and bug assist, (e) for the worst case).** Verified on main: Tauri authority is minted only after `validate_gateway_entry` (`desktop/src-tauri/src/main.rs:2490`), and roster handling never mints. Two gaps remain for step 7: a legitimate node move trips the pinned-origin check (`desktop/src-tauri/src/devserver.rs:722-728`) unless the row lifecycle explicitly clears the pin and re-enters validation, and nothing binds the origin's first label to the entry response's owner and devserver id, so a row-mixing bug or a compromised controller can cross one user's fields with another's tenant label unnoticed. Amendment A14. Found in passing, not a design finding: `is_loopback_gateway_host` treats any DNS name beginning `127.` (for example `127.example.com`) as loopback and waives HTTPS (`devserver.rs:374-384`); it should parse IP literals.

### Observability and operations

**15. The audit trail cannot distinguish a hijack from a reconnect (medium; detection gap under (c) or (d)).** Design line 354's log list omits ownership change. `state.rs` contains exactly four `tracing` sites (boot-id mismatch, command timeout, heartbeat expiry, a ping debug); the `tunnel_up` kill path, every admission decision, and every `issue_kill` emit nothing, and admin kills carry no initiator attribution behind one shared token. After finding 1 or 2 is exercised, controller logs show at most routine heartbeats. Amendment A15.

**16. The aggregate admin API concentrates fleet PII and topology under one fan-out token (low).** `TunnelView.peer_addr` is each devserver user's client IP address, aggregated fleet-wide in one response; `ProxyView` adds package versions and boot ids, useful targeting intelligence. `DEVSERVER_ADMIN_TOKEN` is one value shared by identity, profile, and the operator CLI (`packaging/gateway/scripts/configure.sh:100-106`), two of which are internet-facing services, and the SSE watch endpoints have no connection bound. Amendment A16.

**17. Version agreement is enforced only on the proxy-to-control handshake (low; operational).** Identity, profile, and the admin CLI reach the controller over the plain admin HTTP tree with no version exchange; only release-asset validation (design line 431) guards that skew. Amendment A17.

### Immutable identity and route authorization

**18. A reclaimed username can cross-wire a new account into the prior owner's live devserver (critical; (b)).** Controller and protocol ownership is keyed only by mutable `(user, devserver_id)`: `TunnelRow` has `user: String` but no immutable owner id (`v073/ctl:gateway/crates/devserver-control-proto/src/lib.rs:121-127`), and the actor's `TunnelKey` is `(String, String)` (`v073/ctl:gateway/crates/devserver-control/src/state.rs:19`). Username rename updates only the profile row (`gateway/crates/identity/src/http.rs:788-833`, `gateway/crates/profile/src/http.rs:554-612`); it neither retires the old-name tunnel nor reserves the old handle. The decisive second bug is `devserver_access`: when `caller == owner_id` it returns `role: owner` without checking that `(owner_id, devserver_id)` exists in `devservers` (`gateway/crates/profile/src/http.rs:1189-1222`), contradicting its own unknown-devserver 404 contract. Alice can rename while her tunnel remains registered as `alice`; Bob can then claim `alice`, ask identity to open his devserver, receive Alice's still-live row from the username-indexed controller, pass the unconditional owner shortcut for Alice's devserver id under Bob's user id, and receive an owner entry token to Alice's devserver. A poisoned or compromised controller can exercise the same mismatch without waiting for a rename. A13 and A14 do not catch it: the node base, username label, and devserver id are internally consistent with the poisoned row. Amendment A18.

**19. `viewer` and `editor` imply a security boundary that does not exist; tunnel assertions also fail open (high).** Identity resolves `owner` / `editor` / `viewer` and devserver-proxy copies the role into the gateway assertion, but chan-server admits every request carrying the `TunnelOrigin` marker (`crates/chan-server/src/auth.rs:96-132`). The only authorization use of the role is `TunnelOrigin::owner()` for launcher-management mutations (`crates/chan-server/src/lib.rs:1193-1207`, `crates/chan-server/src/routes/library.rs:410-436`); no route distinguishes viewer from editor. Both non-owner labels therefore mean the same binary thing: this caller has access to the shared devserver. That access includes `GET /api/terminal/ws`, which creates or attaches a PTY, accepts input, and runs as the devserver owner's Unix uid (`crates/chan-server/src/routes/terminal.rs:290-353,587-689`). The shared terminal tenant is deliberately workspace-less and defaults new sessions to `$HOME` (`crates/chan-server/src/devserver.rs:619-636,1417-1421`; `crates/chan-server/src/routes/terminal.rs:314-337`). Keeping unused role labels creates false policy assurance; sharing must instead say plainly that it grants shell-equivalent access to the whole devserver data plane. Ownership is an immutable identity relation, not another role. Separately, a missing or invalid per-request gateway assertion inserts `TunnelOrigin { caller: None }` and continues (`crates/chan-server/src/devserver.rs:1099-1144`), so even the binary boundary fails open. Amendment A19.

**20. Grant revocation, account block, and cookie expiry do not bound established shared access (high; (f) after authorization is removed).** `DELETE /api/grants/{id}` only deletes the profile row (`gateway/crates/identity/src/http.rs:1066-1079`); it never reaches devserver-proxy. The proxy session is a stateless JWT with a 24-hour expiry and no server-side revocation (`gateway/crates/devserver-proxy/design.md:175-177,239-247`). Blocking or deleting a caller evicts tunnels they own, but does not invalidate that caller's cookies or WebSockets on devservers owned by other users. Worse, expiry is checked only during the HTTP upgrade: the WebSocket bridge has a 300-second *idle* timeout and no absolute authorization deadline, so a terminal or document socket kept active by ordinary heartbeats can survive beyond the cookie's `exp` indefinitely. A former grantee can therefore retain file, document, and shell access long after the owner is told the binary grant is gone. A10 only retries tunnel kills and does not address caller sessions. Amendment A20.

### Browser and user-content isolation

**21. Cross-site WebSocket hijacking turns a sibling tenant into owner-context terminal access (critical; (g)).** devserver-proxy requires the CSRF mirror only for `POST`, `PUT`, `PATCH`, and `DELETE`, then sends every WebSocket upgrade down a separate branch after cookie authentication (`gateway/crates/devserver-proxy/src/proxy.rs:245-354,587-630`). It never checks `Origin`. Browser WebSocket handshakes are credentialed GETs and sibling `*.p1.usr.chan.app` origins are same-site, so JavaScript served by Mallory's tenant can open `wss://alice--disc.p1.usr.chan.app/api/terminal/ws` in a victim browser that holds Alice's host cookie. The browser supplies the victim cookie and an attacker origin; the proxy ignores the origin, upgrades, creates or attaches Alice's PTY, and relays bidirectional output and input to Mallory's script. The same class reaches document, scene, event, and library-watch sockets. RFC 6455 defines the `Origin` header for exactly this server-side decision; cookie authentication is not a substitute. Amendment A21.

**22. The credentialed tenant origin can execute user-controlled active content and can be framed by another tenant (high; (f) or malicious repository content).** The file API serves image-classified bytes inline using the extension-derived MIME type; SVG is returned as `image/svg+xml` with no content disposition or sandbox (`crates/chan-server/src/routes/files.rs:304-320,560-650`, `crates/chan-server/src/static_assets.rs:330-359`). No chan-server or devserver-proxy response layer sets CSP, `frame-ancestors`, `X-Content-Type-Options`, or a referrer policy. A grantee can upload an active SVG and induce the owner to navigate to its same-origin file URL; top-level SVG supports script, which then runs beside the owner's gate cookie and readable CSRF cookie. Separately, an attacker tenant can frame an authenticated victim tenant and drive owner actions through clickjacking. The gate's `aud` binding is intact in both cases: the attacker is abusing execution *inside* the correctly bound origin. Amendment A22.

**23. Entry URLs put durable user PII into a signed but unencrypted query token (medium; privacy).** Identity copies caller `name` and `email` into the entry JWT (`gateway/crates/identity/src/http.rs:1341-1353,1453-1465,1668-1680`; `gateway/crates/gateway-common/src/devserver_gate.rs:44-74`) and places that JWT verbatim in `?t=` (`gateway/crates/identity/src/http.rs:1365-1369`). JWT signing provides integrity, not confidentiality: browser history, reverse-proxy request logs, crash reports, and any same-origin request log that captures the URI retain decodable name and email after the 30-second bearer expires. The current design acknowledges entry-token leakage through referers and logs only as a short authorization window (`gateway/crates/devserver-proxy/design.md:175`), but PII does not expire out of a log. No response sets `Cache-Control: no-store` or `Referrer-Policy: no-referrer`. Amendment A23.

## Desired contract

Amend the design as follows before step 5 begins. Each amendment names the design section it changes. A1 through A4 and A18 through A21 are load-bearing; most other amendments assume them.

**A1 (Synchronous fleet admission): an `AdmissionRequest` must be cryptographic evidence of a validated PAT.** Identity's `/internal/v1/tokens/validate` response gains a short-lived, purpose- and version-bound admission lease signed with an identity-only asymmetric key. Its claims bind `owner_user_id`, devserver id, registration id, proxy id, issued time, and expiry; the validate request gains registration and proxy ids. The proxy relays and retains the opaque lease, while the controller holds only the public verification key and verifies its signature, purpose, expiry, and session proxy id before creating a pending claim. Identity re-verifies the same lease when consuming a controller row under A13, so a compromised controller cannot invent an admission that identity will turn into a browser entry. The live devserver client refreshes the lease through PAT revalidation before expiry; the proxy must not retain a raw PAT as a renewal credential. Capability (c) can then fabricate no admission, and (d) is confined to keys whose PATs were actually presented to that node. Finding 1's cap-poisoning and claim-flooding variants die with the same change. A symmetric MAC shared with the controller or proxy does not constrain that verifier after compromise and is not an acceptable endpoint.

**A2 (Trust and network boundaries / Control transport): per-proxy credentials with an individual kill switch.** Check the credential after `ClientHello` against the claimed id, or bind the id into the connect path. Preferred endpoint: a provisioned `{proxy_id: credential}` map or per-node mTLS identity, because either supplies an allowlist and immediate per-node revocation. A derived token, `expected(p) = HMAC(proxy_master_secret, p)`, verified constant-time, is acceptable only with a runtime denylist or equivalent node-disable control; derivation alone limits impersonation to one id but cannot revoke that id independently. Document a two-value overlap window so rotation needs no fleet-wide admission outage, and add the DNS hygiene rule to the public host contract: never publish `*.{id}.` records for unprovisioned ids, and delete them at decommission (finding 2's stale-record chain).

**A3 (Controller state and readiness): anti-displacement and deferred retirement, hard-dependent on A9.** Refuse to displace a heart-healthy incumbent: a duplicate connect for an `Active`, ping-responsive id is rejected instead of honored, and legitimate restarts rejoin after TCP death is detected (clean close immediately, heartbeat within 15 seconds, both already implemented). Until A9 lands, the rejection must not be a `Shutdown`, because `Shutdown` exits the proxy process today (finding 7) and the race is routine: both sides run the same 15-second dead window, so a proxy redialing after a control-path stall would be told to die with all of its tunnels. Use a distinct retryable close frame, or sequence A9 first. When replacement does proceed, keep the incumbent's rows published, marked superseded, until the replacement's snapshot completes or its 30-second deadline fires, then retire the loser. Sessions are stored one per proxy id today, so holding an incumbent alongside a joining replacement is a structural change to session storage, not a staging tweak; the implementation round sizes it. One residual is accepted and named: the predicate protects only `Active` incumbents, so a per-id replacement hold-down plus operator alert covers repeated displacement of a `Joining` session. After A3, a boot-id mismatch is unambiguous attack or fault signal: alert on it, do not just log it.

**A4 (the accepted ruling plus restart reconciliation): land the ruling first; require admission leases and fail closed on duplicates.** The ruling at design lines 37-46 is confirmed security load-bearing and stays the first code of the next round. Every snapshot row carries the unexpired A1 admission lease for the same immutable owner, devserver, registration, and proxy tuple; an absent, expired, malformed, or mismatched lease rejects the row, so initial convergence no longer accepts arbitrary self-authored rows. A live devserver must refresh its lease before expiry to remain eligible for restart restoration. Initial restart reconciliation kills *all* copies of a cross-proxy duplicate key and lets the real client reconnect instead of electing a lexicographic winner whose sort keys are attacker-chosen. Within one controller lifetime, a non-empty snapshot under a boot id different from the last boot id seen for that proxy is quarantined and alerted; a fresh proxy process has an empty registry. This needs a small bounded proxy-id-to-boot-id memory that survives session removal and is exempt only during initial convergence, where lease verification remains mandatory. Reconciliation loser kills, especially capacity trims, get a distinct warn-level line that operators alert on. Residual trust is bounded and explicit: a compromised node can replay a lease for a user whose PAT it actually observed only until lease expiry, and a compromised controller can suppress rows or issue denial-of-service kills but cannot forge the identity signature that identity rechecks before minting.

**A5 (Control transport / Controller state and readiness): numeric and shape bounds.** Bound every string and identifier before allocation or insertion, including username, devserver id, version, origin, and error text; reject non-canonical values at the protocol edge. Bound cumulative snapshot bytes (for example 16 MiB) in addition to rows; apply the per-session row cap to live rows inclusive of deltas; cap pending claims per session and globally; cap aggregate fleet rows, bytes, sessions, and SSE watchers; coalesce watch publication to the 1-second tick behind a dirty flag instead of after every command. The verification suite must exercise just-below, exact-limit, and just-above cases for every bound.

**A6 (Trust and network boundaries): per-node entry and assertion keys.** Derive separate domain-separated keys for entry-token verification and proxy-to-devserver assertions, for example `HMAC(gate_master, "entry" | proxy_id)` and `HMAC(gate_master, "assertion" | proxy_id)`; identity already learns `proxy_id` from the aggregate row and derives only that node's entry key. Each proxy holds only its own derived keys. A20 replaces the fleet-valid session JWT with an opaque proxy-local session, so no proxy needs a key capable of minting a portable browser session. Master rotation still rotates everything, but capability (d) degrades to the node's inherent data-path power instead of fleet-wide cookie minting.

**A7 (Trust and network boundaries / Packaging): make confidential transport load-bearing for all user and control data.** Raw PATs, internal bearers, file contents, terminal input and output, document and scene updates, WebSocket frames, entry assertions, and control traffic must be encrypted and integrity-protected on every non-loopback leg. Require WireGuard or equivalent authenticated overlay, or TLS with operator-supplied certificates; plain h2c/HTTP is acceptable only on a verified loopback hop. Split identity's `/internal/v1/*` tree onto a second listener bound to the protected interface, bind every internal listener to its provisioned interface instead of `0.0.0.0`, and express the same restriction in host firewall and container-network policy. Packaging verification must prove the public interface cannot reach internal listeners and must capture the effective scheme and peer identity for each hop. Document token and certificate rotation order. Carried unverified: the repository ships no edge or nginx configuration, so today's `/internal` exposure rests on the merged router plus whatever edge the deployment runs.

**A8 (Controller state and readiness): observed kill completion plus tombstones.** A successful `CommandResult` records intent but does not remove the aggregate row. The controller removes it only after the same session emits the matching contiguous `TunnelDown` for that registration UUID; timeout or sequence gaps force resync and keep the row unavailable for new entries. Keep a bounded memory-only FIFO of completed registration UUIDs, per proxy id, with expiry at least as long as the maximum browser-session lifetime. A later snapshot row or `TunnelUp` carrying a tombstoned UUID proves the proxy lied: refuse the snapshot, quarantine the node, and page an operator. A fully compromised proxy can still lie consistently by hiding the tunnel while serving it, so the ADR must name node isolation or deprovision as the only definitive response.

**A9 (Proxy failure semantics): do not die on `Shutdown`.** A hello rejection or `Shutdown` during connect means retry with long capped backoff while staying unready: no new admissions, existing tunnels keep serving, operator alerted. Reserve process exit for local configuration faults. This is one branch in `supervise`, and it removes both the guaranteed fleet-wide tunnel kill on every lockstep upgrade and the (e)-triggered mass exit, at the price of stale binaries lingering unready during a botched rollout, which is the safer failure direction.

**A10 (Identity, profile, and admin changes): bounded revocation retry.** On `Err`, timeout, or partial completion from either a tunnel kill or A20 session revoke, identity and profile retry with exponential backoff over roughly five minutes and record final failure in `auth_audit`, which both services already write. Grant delete revokes sessions for that subject, owner, and devserver tuple; account block or delete revokes the subject fleet-wide before returning success. APIs may return accepted/pending while retrying, but must not report completed revocation until the controller has observed every reachable node's confirmation. Absolute session and bridge deadlines remain the backstop for an unreachable or lying node.

**A11 (Public host and routing contract): `__Host-` cookie names at the cutover.** Rename the production cookies exactly to `__Host-devserver_gate`, `__Host-devserver_csrf`, and `__Host-id_session`, and update Chan Desktop's hard-coded cookie allowlist and all tests. Production setters must always emit `Secure; Path=/` with no `Domain`; insecure local development must use HTTPS or visibly distinct test-only names because browsers correctly reject `__Host-` cookies without `Secure`. File a PSL submission only after testing the actual rule depth: an entry for `usr.chan.app` alone makes each node host such as `p1.usr.chan.app` registrable but does not make sibling tenant origins below that node cross-site. If site partition is desired as defense in depth, the PSL rules must treat every node base as a suffix, without replacing exact-origin checks in A21. v0.74.0 is the exact-version cutover, so the rename rides it cleanly.

**A12 (Synchronous fleet admission): registration-transfer damping.** Reject or quarantine a key whose authoritative proxy or registration changes repeatedly within the last N seconds, a small actor rule on the `Instant`s the controller already holds; immutable ownership itself cannot change under A18. It is orthogonal to the live-first joining ruling, which governs a different path. Document PAT rotation and A20 session revocation as the remediation for theft.

**A13 (Identity, profile, and admin changes): identity re-anchors node bases, ownership, and admission.** Before minting, identity enforces that the row's node base host is exactly one DNS label under `DEVSERVER_PROXY_ORIGIN` with matching scheme and effective port, the same string check the desktop performs, so design line 160's rule becomes self-enforcing instead of delegated to the controller. It re-verifies the row's A1 admission lease, requires every lease claim to match the controller row, and separately requires the immutable owner and devserver to match the profile record. Username, label, and devserver id consistency alone is insufficient.

**A14 (Chan Desktop discovery and exact-origin validation): bind identity, label, and pin.** The entry response and token carry immutable `owner_user_id`; the entry validator requires it to match the roster row and requires the origin's first label to equal `{username}--{devserver_id[..12]}` lowercased. The move-node lifecycle explicitly clears only the per-row origin pin, preserves the immutable owner and devserver binding, and re-enters through `validate_gateway_entry` with `pinned_origin = None`. Fix `is_loopback_gateway_host` to parse IP literals instead of prefix-matching `127.`.

**A15 (Proxy failure semantics / logging): make hijacks audible without leaking credentials.** Extend design line 354's list: every ownership change at WARN with immutable owner id, display username, devserver id, old and new proxy id and boot id, and the control peer address; every admission denial and every cross-session `Stale`; every admin kill, revoke, and read with initiator and caller peer address. Document the flap pattern, one key changing nodes within seconds, as the hijack signature operators alert on. Entry credentials, PATs, cookies, assertions, authorization headers, query strings, names, email addresses, and file or terminal content are always redacted.

**A16 (Aggregate admin API): least-privilege service identities, PII, and watcher bounds.** Replace the fleet-wide admin bearer shared by identity, profile, and operators with distinct rotatable client credentials or mTLS identities scoped to the exact read, kill, revoke, or watch operations each caller needs. Identity and profile must not inherit operator fleet-list access. Document `peer_addr` as PII, keep it on authorized per-user detail reads rather than the fleet list, redact it from routine logs, and bound concurrent SSE watchers per identity.

**A17 (Aggregate admin API): version header.** `/admin/v1/*` responses carry the controller's `package_version`; `DevserverControlClient` logs or refuses on mismatch, extending the immutable-version acceptance criterion (design line 523) to the consumer leg.

**A18 (Public host contract / Control protocol / Identity and profile changes): immutable owner binding end to end.** Add `owner_user_id` to PAT validation, the signed admission lease, `TunnelRow`, `AdmissionRequest`, controller keys and views, entry claims, gate assertions, and Desktop roster validation. The authoritative key is `(owner_user_id, devserver_id)`; username is routing and display metadata only and may never grant access. Profile's owner shortcut first proves the exact devserver row exists and belongs to that owner. Identity selects controller rows by owner id and rejects any profile/controller mismatch. Username change retires old-name controller rows and A20 sessions before the old handle can be reused; exact-cutover deployments should prefer discriminated hosts and remove the bare username host until non-ambiguous reuse is proven. Tests must cover rename, immediate handle reclaim, stale rows, and a controller that deliberately mixes valid fields from two users.

**A19 (Identity, profile, proxy, and chan-server authorization): one binary devserver grant, no devserver-share roles.** Remove `devserver_grants.role` and the corresponding role input and output from devserver-share APIs, access decisions, Desktop roster rows, entry and session state, and gateway assertions. This does not change any separate workspace-sharing model. A devserver grant's existence authorizes one subject to the exact immutable `(owner_user_id, devserver_id)` data plane; its absence does not. The contract explicitly includes workspace data, document and scene mutation, terminal, process, and arbitrary-path access exposed by that devserver, so the share UI and confirmation must call it shell-equivalent access. Ownership is derived only from `subject_user_id == owner_user_id`. Gateway control-plane operations outside the shared data plane, including changing grants and registration metadata, remain owner-only. A chan-server-local owner check may remain as product behavior, but it is not a containment boundary because a grantee with terminal access can mutate the same local state. A missing, malformed, expired, wrong-audience, wrong-devserver, or wrong-owner assertion fails closed before any tunnel route. Put the binary grant check in shared route middleware.

**A20 (Identity, profile, controller protocol, and proxy sessions): revocable proxy-local sessions.** After a per-node entry token is exchanged, devserver-proxy creates a cryptographically random opaque cookie and stores a bounded in-memory record containing subject id, immutable owner id, devserver id, creation time, absolute expiry, and cancellation handle. Proxy restart clears the records fail closed. Add a `RevokeSessions` command keyed by `(subject_user_id, owner_user_id, devserver_id)` and a subject-wide form. The controller fans each revoke to every active proxy because a prior node can retain an established stream after a tunnel transfer; grant delete, account block, and delete invoke it. Proxies delete matching records and actively close HTTP streams and WebSocket bridges before acknowledging, and a joining proxy starts with no sessions. Session lookup revalidates the record on every request. Enforce short absolute session and bridge lifetimes independent of activity as a backstop, with the controller retries in A10. Do not use another long-lived stateless bearer as the session store.

**A21 (Proxy browser boundary): exact-origin WebSocket authorization and absolute deadlines.** Every cookie-authenticated WebSocket upgrade requires one canonical `Origin` equal to the externally visible request origin after trusted-proxy normalization; forwarded host and scheme headers are honored only from configured trusted edge peers. Missing, multiple, opaque, wrong-scheme, wrong-port, sibling-tenant, and malformed origins return 403 before reaching chan-server. This applies to terminal, document, scene, event, library-watch, and future upgrades. Browser-facing state-changing `GET` and subscription setup are classified as unsafe and receive equivalent origin/CSRF protection, including the binary devserver authorization check where a socket carries mutations. Pass the session's absolute expiry and A20 cancellation handle into the bridge and close it when either fires, even under continuous traffic. Tests use real browser-shaped handshakes, not a client that omits `Origin` by default.

**A22 (Proxy and chan-server browser response policy): contain active content and ambient authority.** Authenticated application responses set `Content-Security-Policy` with `frame-ancestors 'none'`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: no-referrer`, and `Cache-Control: private, no-store` where content is user-specific or carries authorization state; no credentialed wildcard CORS is permitted. User-controlled active formats, including SVG and HTML, are either served from a cookieless untrusted-content origin or forced to attachment with a restrictive `Content-Security-Policy: sandbox` and no ambient credentials. The v0.74 minimum is sandbox plus attachment for active formats and framing denial for every credentialed tenant page. Generated application assets receive an allowlisted MIME type rather than extension-only trust.

**A23 (Entry handoff and privacy): no PII or bearer in a logged URL.** Remove name and email from all authorization tokens; resolve optional display identity through an authenticated internal lookup after exchange. Deliver the short-lived entry credential only in a POST body, using an identity-hosted auto-submitting form for browser navigation and a direct POST from Chan Desktop; the browser exchange requires the canonical identity `Origin`, never accepts a query or fragment credential, and returns a clean 303. Give each entry a random `jti` and reject replay through a bounded proxy-local cache retained until expiry. Entry, exchange, and redirect responses set `Cache-Control: no-store` and `Referrer-Policy: no-referrer`; access logs redact request bodies and authorization material. Tests assert that browser-visible URLs, history fixtures, proxy logs, audit rows, and error reports contain neither the token nor decoded PII.

## Verified sound, kept out of the findings

The review tried and failed to break these: stale incarnations cannot mutate or disconnect a replacement session (`require_key`, with direct test coverage including a retired connection's late frames, disconnects, and command results); both bearers are compared constant-time; a proxy cannot register an off-template origin (finding 2 is about which on-template id, not arbitrary origins); every existing internal queue is bounded and closes the session on overflow rather than dropping state; the 30-second snapshot deadline is absolute across the whole phase; the 10-second admission wait, `AdmissionCancel`, and 15-second claim expiry are implemented as specified; kills target registration UUIDs, so delayed commands cannot hit successors; controller reads fail closed at 503 while unready; PAT guessing is throttled at both proxy and identity; gate cookies bind `aud` and `drv` per request, so transplanting one tenant's cookie to another host or devserver fails, although finding 21 shows that an unchecked WebSocket origin lets a victim browser use its own correctly bound cookie on an attacker's behalf; HS256 is hard-required with `alg:none` rejected and `typ` dispatched; `owner` and `disc` label validation forbids dots and lookalikes; packaging keeps secrets out of images and examples.

The accepted joining ruling and the step-4 barrier evidence were review inputs, not targets; the only addition to them here is A4's residual list.

## Boundaries

- Every amendment fits the design's existing shape: in-memory singleton controller, no controller database, no controller HA, no proxy-to-proxy path, no mixed-version support, no automatic DNS or certificate provisioning. A20 deliberately adds bounded in-memory authorization-session state to each proxy; it does not add durable control-plane state. The explicit non-goals at design lines 525-532 stand.
- This document proposes spec-level amendments only; it changes no code. Implementation lands in the step 5 to 9 rounds under the design's red-before-green discipline.
- The findings grade the design, not the `v073/ctl` implementation, which implements the design as written; most findings are properties of the design text.
- Carried unverified, honestly: the server-side duplicate-cookie resolution order behind the `id_session` fixation sub-claim (A11 closes the class regardless); whether any deployment edge blocks `/internal/*`; the deployment binding of the proxy h2c listener; and the edge's forwarded-header normalization policy. A7 and A21 make all four explicit deployment tests.

## Acceptance

- A1 through A4 and A18 through A21 land before broad step-5 consumer migration, with A3's duplicate-connect rejection shipped only alongside A9 or as a retryable close frame; A4's ruling lands first as already scheduled.
- Each amendment is reflected in the named design-doc section, the relevant identity, proxy, chan-server, and Desktop design docs, and ADR `gateway/docs/adr/0002-control-plane-owns-proxy-fleet-state.md`. The ADR names the residual trust honestly: restart reconciliation accepts only identity-signed admissions but trusts their proxy liveness claim until lease expiry, a consistently lying proxy requires node isolation, revocation against an unreachable node is bounded by absolute expiry, and the protected network is load-bearing for confidentiality of user as well as control data.
- Control-plane verification proves that a fabricated `AdmissionRequest` plus `TunnelUp` without a lease is refused and the prior owner is never killed; a snapshot with an absent, expired, malformed, replayed-on-another-node, or field-mismatched lease is rejected; identity independently rejects a controller row with a forged or mismatched lease; a duplicate connect against a healthy incumbent is rejected; a changed boot id outside initial convergence is quarantined; `CommandResult` alone does not remove a row; a tombstoned UUID reappearing quarantines the node; and every field, session, row, byte, claim, fleet, and watcher bound has boundary tests under paused Tokio time where applicable.
- Identity verification proves that username rename plus immediate handle reclaim cannot expose the prior owner's live devserver; the owner shortcut returns 404 for an unknown devserver id; controller/profile owner-id mismatch fails closed; and sending a valid cookie, entry, or assertion to the wrong host, node, owner id, or devserver id never returns tenant data.
- Authorization verification proves no devserver-share role field remains in `devserver_grants`, the devserver-share APIs, tokens, sessions, assertions, or roster rows; an owner is derived from immutable subject equality; a grantee reaches only the exact shared devserver data plane, including its terminal; an ungranted caller and a grant for another devserver fail closed; gateway grant and registration operations remain owner-only; no chan-server-local owner check is claimed as isolation from a shell-equivalent grantee; and missing or invalid assertions fail closed.
- Revocation verification proves grant delete terminates matching proxy sessions and active WebSockets, block and delete terminate the subject fleet-wide, retries survive partial node failure, proxy restart loses sessions fail closed, and continuously active bridges close at their absolute deadline.
- Browser verification proves missing, sibling, malformed, and wrong-scheme or wrong-port WebSocket origins return 403 while the exact canonical origin works; active SVG or HTML cannot execute with tenant credentials; credentialed pages cannot be framed; authenticated dynamic responses are non-cacheable and nosniff; and no credentialed wildcard CORS is emitted.
- Privacy and deployment verification proves entry URLs, logs, history, audit rows, and errors contain neither bearer nor decoded PII; internal listeners are unreachable from public interfaces; and all non-loopback hops carrying PATs, cookies, assertions, file data, terminal data, or WebSocket frames are encrypted and peer-authenticated.
- A multi-node isolation matrix runs at least two owners plus one granted and one ungranted caller across three proxies, including concurrent same-devserver-id values under different owners, wrong-node cookies, username rename and reclaim, grant deletion during live sockets, proxy restart, and one partitioned node. No response, stream, roster, audit record, or reconnect crosses immutable owner or exact-grant boundaries.
- No amendment expands scope into the design's non-goals, and the version cutover stays exact.
