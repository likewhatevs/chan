# Fold Chevrons Appear Next To `#` Comments Inside Fenced Code Blocks

Status: accepted scope for v0.74.0. A confirmed defect with a user report.

## Problem

In the WYSIWYG editor, a fenced code block that contains shell-style comments gets a fold chevron in the gutter next to every one of those comment lines. A ```bash fence whose body starts with `# install deps` renders a chevron beside that line exactly as if it were a level-1 markdown heading. The owner reported it from a shell script; the same thing happens in every fenced language with `#` line comments, which is shell, Python, YAML, TOML, Dockerfile, Ruby, Perl, Makefile and nginx config.

The mechanism is one regex applied to raw document lines. `web/packages/workspace-app/src/editor/fold.ts:26` declares `const HEADING_RE = /^(#{1,6}) /;` and `headingLevel(text)` at `web/packages/workspace-app/src/editor/fold.ts:29-32` runs it against a plain string. Nothing in the file's 153 lines imports `syntaxTree`, tracks a fence marker, or looks at a node name; the imports are `codeFolding, foldEffect, foldedRanges, foldService, unfoldEffect` from `@codemirror/language` plus `gutter, GutterMarker` from `@codemirror/view` (`web/packages/workspace-app/src/editor/fold.ts:16-24`). There are three fence-blind consumers, and the fix has to reach all three because the range logic is duplicated rather than shared: the fold service at `web/packages/workspace-app/src/editor/fold.ts:34-50`, the gutter `lineMarker` at `web/packages/workspace-app/src/editor/fold.ts:88-101` whose single `headingLevel(line.text) === 0` test is the entire chevron decision, and the gutter click handler at `web/packages/workspace-app/src/editor/fold.ts:116-143`, which re-derives the forward walk inline at `web/packages/workspace-app/src/editor/fold.ts:130-137` with the comment "re-derive" because the service's range is not reachable from there.

This is a known trade whose cost was not fully paid, not an accident. The file's own header at `web/packages/workspace-app/src/editor/fold.ts:1-14` states why the gutter is hand-rolled: `@codemirror/lang-markdown` attaches `foldNodeProp` to paragraphs, blockquotes, tables and fenced code, so the stock `foldGutter()` renders a chevron on nearly every line, and `foldGutter` has no per-line callback to filter down to headings. Replacing it was correct. What was dropped along with it is the language awareness that came free with the tree-driven folder, and that awareness was never rebuilt on the replacement. The spec records the same gap: `web/packages/workspace-app/src/editor/design.md:43` defines fold as "line `^#{n} ` folds end-of-line -> start of next `#{<=n}` line (or doc end)" with no fenced-code or frontmatter exclusion, so the implementation matches its written spec and a future reader restoring the spec restores the bug.

The damage runs past the stray chevron, in two directions, both of which produce fold ranges that markdown block structure never sanctions.

Clicking the bogus chevron collapses to end of document. The click handler treats `# install deps` as level 1 and scans forward for the next line with level <= 1 (`web/packages/workspace-app/src/editor/fold.ts:130-137`). A following `## Next section` is level 2 and `### Deeper` is level 3, so neither stops the walk. It falls through to the doc-end fallback (`web/packages/workspace-app/src/editor/fold.ts:49` in the service, `foldTo = view.state.doc.length` at `web/packages/workspace-app/src/editor/fold.ts:127` in the handler) and hides the rest of the code block, the closing fence, and every subsequent section. Because level 1 is the most common shell-comment shape, the worst case is also the default case.

A real heading's fold is truncated by the fence, with no bogus click involved. Folding a genuine `## Setup` whose body contains a ```bash block stops the forward walk at the fenced `# install deps`, since level 1 is <= every level. The resulting range hides the prose and the OPENING fence line while leaving the fence body and its closing fence visible. The user collapses a section and is left looking at orphaned code with no fence around it.

Frontmatter is the same defect on a second surface. `web/packages/workspace-app/src/editor/markdown/frontmatter.ts:1-11` exists precisely because "a `# ` inside the YAML body would be promoted to a heading", and it isolates the block into `Frontmatter` / `FrontmatterMark` nodes so the parser never sees one. `fold.ts` bypasses the parser, so a commented frontmatter (`# deck config` above a `chan:` block, the common shape for slide decks) still gets a chevron and can collapse the whole document from inside the frontmatter.

Two adjacent sites share the missing-fence-awareness class and belong in the same item. `web/packages/workspace-app/src/editor/commands/format.ts:129` (`/^((?:[#]+\s+)|(?:>\s+))/` in `setBlockKind`) and `web/packages/workspace-app/src/editor/commands/format.ts:421` (the `|(?:[#]+\s+)` alternative in `toggleLinePrefix`) strip a leading `#` run on the caret line with no fence check, so a bullet-list chord with the caret on a fenced `# install deps` rewrites it to `- install deps` and the h2 chord rewrites it to `## install deps`. That destroys code rather than only misrendering it, and it fires only on an explicit formatting command. `web/packages/workspace-app/src/editor/slides.ts:157` (`/^#{1,6} /` in `firstSlideHeadingCaret`, `web/packages/workspace-app/src/editor/slides.ts:151-163`) is fence-blind too, but its sole caller is `landCaretAtFirstHeading` at `web/packages/workspace-app/src/state/commands/slides.ts:34`, which runs against a freshly written server seed whose first body line is `# Slide 1`; worst case today is a caret one line off on deck creation, and it becomes real the moment the helper is reused on user-authored content.

The brief's suspicion that slide splitting shares the bug is refuted. `splitSlidePages` (`web/packages/workspace-app/src/editor/slides.ts:104-126`) and `slidePageBreakLines` (`web/packages/workspace-app/src/editor/slides.ts:207-216`) key only on `PAGE_BREAK_RE` (`web/packages/workspace-app/src/editor/slides.ts:26-27`, the `<hr class="chan-page-break">` and `@pagebreak` forms) and never look at `#`. `groupHeadingsBySlides` (`web/packages/workspace-app/src/editor/slides.ts:83-102`) buckets headings it is handed, and its only caller passes the fence-aware outline output (`web/packages/workspace-app/src/components/OutlineBody.svelte:47`). A fenced comment cannot start a slide.

The outline is likewise clean, which is what makes the fold gutter a lone outlier rather than a systemic gap. `parseHeadings` at `web/packages/workspace-app/src/components/OutlineBody.svelte:60-89` tracks `inFence` / `fenceMarker` and skips fenced lines before matching. `web/packages/workspace-app/src/editor/extensions/wikiBlocks.ts:14-15` pairs its `HEADING_RE` with a `FENCE_RE` guard. `web/packages/workspace-app/src/demo/search.ts:98` guards with `!inFence`. `web/packages/workspace-app/src/editor/decorations/headings.ts:58-63` and `scrollToHeading` at `web/packages/workspace-app/src/editor/Wysiwyg.svelte:874-877` key off `ATXHeading1..6` nodes. The Rust side does it too (`crates/chan-workspace/src/markdown/headings.rs:1-4` opens by naming this exact hazard). So the outline sidebar and the fold gutter currently disagree about what a heading is in the same open document.

Scope is bounded: `headingFold()` is installed only in WYSIWYG (`web/packages/workspace-app/src/editor/Wysiwyg.svelte:56`, `web/packages/workspace-app/src/editor/Wysiwyg.svelte:490`); Source mode has no fold gutter. Folding is a view decoration, so there is no data loss and a second click undoes it, but there is no escape hatch either: the package registers no `foldKeymap`, `foldAll` or `unfoldAll`, so a user who folds to document end and scrolls away must find that one chevron again.

`fold.ts` has no test file. There is no `editor/fold.test.ts`, no test imports `headingFold` (its only two referencing files are its own definition and `Wysiwyg.svelte`), and no `?raw` source pin covers it. Nothing today asserts fold ranges, chevron placement, the level-comparison walk, the doc-end fallback or the click toggle.

## Desired contract

Heading detection in the editor has one source of truth: the lezer syntax tree. A line is a heading if and only if the tree resolves it to `ATXHeading1..6`. Stated once, applied by every consumer, so a fenced or frontmatter `#` cannot be mistaken for a heading in a third place later.

Concretely, `fold.ts` stops calling a regex on raw line text. The chevron decision, the fold-range walk in the service, and the duplicated walk in the click handler all read the tree. The natural shape is to collect heading positions and levels once from a tree iterate and have all three consumers read that collection, which also removes the duplicated forward walk that currently makes the file fixable in one place and broken in another.

Fixing this via the tree covers fenced code, indented fenced code, tilde fences, inline code spans and frontmatter in one change. A hand-rolled fence tracker covers only fenced code and would leave frontmatter broken, so it is not an acceptable substitute.

The lazy-parse caveat must be respected. `syntaxTree(state)` is viewport-budgeted, and `web/packages/workspace-app/src/editor/decorations/walker.ts:145-157` documents this and reaches for `ensureSyntaxTree(state, to, PARSE_BUDGET_MS) ?? syntaxTree(state)`. The gutter's `lineMarker` only asks about visible lines, so it is safe on the lazy tree. The fold service's forward walk runs to end of document and can leave the parsed region, so the truncation case needs `ensureSyntaxTree` or a collected-ranges approach, not a naive per-line `resolveInner` inside the loop.

The precedent and the helper already exist in the same directory. `web/packages/workspace-app/src/editor/widgets/tag.ts:37-47` solves the identical problem for `#word` hashtag pills with a `SKIP_INSIDE` set covering `InlineCode`, `FencedCode`, `CodeBlock`, `CodeText`, `CodeMark`, `CodeInfo` and friends, collected over the viewport with `syntaxTree(state).iterate`. `web/packages/workspace-app/src/editor/tools.ts:74-88` has a `fencedCodeRanges(view)` of the same shape. `chanMarkdown()` (`web/packages/workspace-app/src/editor/markdown/grammar.ts:33-38`) is installed at `web/packages/workspace-app/src/editor/Wysiwyg.svelte:489`, one line above `headingFold()`, so the nodes are already in the tree, and `fold.ts` already imports from `@codemirror/language`, so this adds a name to an existing import and no dependency.

The formatting chords get the same guard. `setBlockKind` and `toggleLinePrefix` in `web/packages/workspace-app/src/editor/commands/format.ts` must decline the heading-prefix strip when the caret line resolves inside a code node, so a bullet or heading chord inside a fence leaves the comment intact. `firstSlideHeadingCaret` (`web/packages/workspace-app/src/editor/slides.ts:151-163`) is string-only with no `EditorState` in hand, so it takes the cheap fence tracker that `OutlineBody` and `wikiBlocks` already use rather than a tree; that is a consistency fix, not a user-facing one.

`web/packages/workspace-app/src/editor/design.md:43` is rewritten in the same change to state that fold headings come from the syntax tree and that fenced code, indented and tilde fences, inline code and frontmatter never produce a fold chevron. The current wording is the spec that authorizes the bug.

Two behaviour decisions the item must make and record. First, CommonMark allows up to three leading spaces on an ATX heading and `web/packages/workspace-app/src/editor/extensions/wikiBlocks.ts:14` matches that with `/^\s{0,3}#{1,6}\s/`, while `fold.ts` today requires column 0; a tree-driven fold will start folding those indented headings, which is a correctness gain but a visible change and should be called out rather than discovered. Second, decide whether Setext headings (`===` / `---` underlines), which the tree exposes as `SetextHeading1` / `SetextHeading2` and which the regex can never see, are in or out; out is defensible, silence is not.

## Acceptance

Every check names the deliberate mutation that must turn it red, per the standing rule recorded at `team/release/release-v0.73.0.md:37`.

A new `web/packages/workspace-app/src/editor/fold.test.ts` is required. The file has zero coverage today, so a fix without tests ships into an unguarded file.

1. **No chevron on a fenced `#` comment.** A document with a real `# Real Heading`, then a ```bash fence containing `# install deps` at column 0; assert the gutter marker resolves to `null` for the fenced line and non-null for the real heading. Red mutation: restore `headingLevel(line.text)` as the `lineMarker` test at `web/packages/workspace-app/src/editor/fold.ts:97`.
2. **No chevron on a tilde fence.** Same document with `~~~bash` / `~~~` delimiters instead of backticks. Red mutation: replace the tree lookup with a fence tracker whose regex matches only backticks; the backtick test at check 1 stays green and this one reddens, which is exactly why both exist.
3. **No chevron inside an indented fence.** A ```` ``` ```` fence indented by two spaces inside a list item, containing `# comment` at the fence's own indentation and a second copy at column 0. Today neither fires, because `HEADING_RE` is anchored with zero leading-whitespace tolerance, so this check guards the fix rather than the current bug: a tree-driven implementation must keep both silent while the same change starts folding a genuinely indented `  ## Heading`. Red mutation: strip leading whitespace from the line before deciding, which is the obvious way to add CommonMark indent tolerance and the way that reintroduces the bug inside indented fences.
4. **No chevron on a `#` inside an inline code span.** A paragraph line whose entire content is `` `# not a heading` ``. Today this passes by accident since the backtick occupies column 0; make it fail-proof by also covering a line that is exactly `` ` `` on its own followed by a `# x` line inside a fence-free inline span construct, or assert the node-level rule directly by checking that the skip set includes `InlineCode`. Red mutation: drop `InlineCode` from the skip set used by the fold path.
5. **No chevron on a frontmatter comment.** A document opening `---` / `# deck config` / `chan:` / `  kind: slides` / `---`; assert no marker on the `# deck config` line. Red mutation: guard on `FencedCode` / `CodeBlock` only and omit `Frontmatter`; checks 1 and 2 stay green and this one reddens.
6. **Shebang stays unmatched, and `#` without a space stays unmatched.** Assert no chevron for `#!/bin/bash`, `#!/usr/bin/env bash`, `#foo`, a bare `#`, and `####### seven` (seven hashes), whether inside a fence or in body text. These pass today (the regex needs a literal space and caps at six hashes) and the fix must not widen them. Red mutation: relax the heading test to `/^#{1,6}/` without the trailing space, or accept `#{1,}`.
7. **A `#` comment with a space in body text still folds.** A plain `# Real Heading` and `## Sub` in prose produce chevrons at levels 1 and 2. Red mutation: make the tree lookup reject every line, i.e. any over-broad skip set; this is the counterweight that stops the fix from being "no chevrons anywhere".
8. **A real heading's fold range is not truncated by a fenced comment.** `## Setup`, prose, a ```bash fence containing `# install deps`, more prose, then `## Next`; assert the fold range for `## Setup` ends at `## Next`, not at the fenced comment. This is the check a gutter-only fix would leave broken. Red mutation: fix the `lineMarker` only and leave the forward walk at `web/packages/workspace-app/src/editor/fold.ts:40-46` regex-driven.
9. **The click handler agrees with the service.** Same document as check 8, driven through the gutter click path; assert the dispatched `foldEffect` range equals the service's range. Red mutation: fix `web/packages/workspace-app/src/editor/fold.ts:40-46` and leave the duplicate at `web/packages/workspace-app/src/editor/fold.ts:130-137` regex-driven. This check exists because the duplication is the reason a half-fix is easy.
10. **The doc-end fallback survives.** A trailing `## Last` with body and no following heading folds to document end, and a heading on the final line with nothing after it yields `null` (`web/packages/workspace-app/src/editor/fold.ts:47-49`). Red mutation: return `null` whenever no following heading is found, which silently removes folding for every last section.
11. **Long-document correctness past the viewport.** A document large enough that the lazy tree does not cover it, with the real heading near the top and its terminating heading far below; assert the fold range still reaches the correct terminator. Red mutation: use `syntaxTree(state)` alone in the forward walk instead of `ensureSyntaxTree`, per the caveat at `web/packages/workspace-app/src/editor/decorations/walker.ts:145-157`.
12. **Formatting chords leave fenced comments alone.** With the caret on `# install deps` inside a ```bash fence, the bullet-list chord and the h2 chord both no-op; with the caret on a real `# Heading`, both still rewrite. Red mutation: remove the fence guard from `setBlockKind` (`web/packages/workspace-app/src/editor/commands/format.ts:129`) or from `toggleLinePrefix` (`web/packages/workspace-app/src/editor/commands/format.ts:421`); each half must redden independently.
13. **`firstSlideHeadingCaret` skips a fence.** A deck source whose post-frontmatter content opens with a fence containing `# not a title` before a real `# Slide 1`; assert the returned offset is the end of `# Slide 1`. Red mutation: revert to the bare `/^#{1,6} /` at `web/packages/workspace-app/src/editor/slides.ts:157`.
14. **Spec and implementation agree.** `web/packages/workspace-app/src/editor/design.md:43` states the tree-driven rule and its exclusions. Verified by review, not by a test.
15. **Gate.** `npm run check` plus the full vitest run in `web/`, not just the focused file. Re-run any red on the heavy async-import tests in isolation before treating it as a regression.
16. **Owner hand-check.** The originally reported document open in WYSIWYG: no chevrons beside the shell comments, a real heading above the fence folds past the whole fence, and the outline sidebar and the gutter now name the same set of headings.

## Boundaries

In scope: `web/packages/workspace-app/src/editor/fold.ts` in full, including the duplicated range walk in the click handler; the heading-prefix strip in `web/packages/workspace-app/src/editor/commands/format.ts`; `firstSlideHeadingCaret` in `web/packages/workspace-app/src/editor/slides.ts`; spec item 9 in `web/packages/workspace-app/src/editor/design.md`; and a new `web/packages/workspace-app/src/editor/fold.test.ts`.

Out of scope: the decision to hand-roll the gutter instead of using `foldGutter()`, which stays as it is for the reason given at `web/packages/workspace-app/src/editor/fold.ts:1-14`; the outline, wiki-block, demo-backend, heading-decoration and Rust heading paths, all of which already track fences correctly; slide page splitting, which keys on `PAGE_BREAK_RE` and never looked at headings; `web/packages/workspace-app/src/editor/widgets/tag.ts` and `web/packages/workspace-app/src/editor/tools.ts`, which are the model rather than the target; Source mode, which installs no fold gutter; and adding `foldAll` / `unfoldAll` / a fold keymap, which is a real gap in the recovery story but a separate feature.

One test-fidelity note that does not require code change: `testHeadings` at `web/packages/workspace-app/src/editor/slides.test.ts:291-300` reimplements the heading match without a fence guard, so the slides grouping tests feed `groupHeadingsBySlides` heading lists that the fence-aware production outline would never produce. Worth a comment there, not a fix.
